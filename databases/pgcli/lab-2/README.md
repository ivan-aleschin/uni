## 1. Резервное копирование (Backup) и восстановление (Restore)

Поскольку в 3-й лабораторной работе мы будем модифицировать структуру базы, нам необходимо уметь делать бэкап и откатывать `demo` до изначального состояния, чтобы ничего не сломать. 
Используем стандартные утилиты PostgreSQL, которые работают в терминале (в том числе в `fish` на NixOS).

### Как сделать бэкап (сохранить изначальное состояние)

Перед тем как запускать скрипты из 3-й лабы, сделайте снимок текущей рабочей базы в файл `demo_backup.dump`:

```bash
# Выполнять в обычном терминале, находясь в папке с репозиторием
mkdir -p backups
pg_dump -h localhost -U $USER -F c -f backups/demo_backup.dump demo
```
* `-F c` — формат "custom", удобный для полного восстановления.
* `backups/demo_backup.dump` — файл нашего эталонного бэкапа, мы его **не перезаписываем**, пока нам нужна изначальная БД.

### Как восстановить базу (Откат)

Если скрипты отработали не так, как ожидалось, или вам снова нужна чистая база `demo` для других лабораторных, восстановите её. 
⚠️ **Внимание:** перед откатом убедитесь, что вы вышли из `pgcli` и нет других активных подключений к `demo`.

```bash
# 1. Удаляем сломанную/измененную базу
dropdb -h localhost -U $USER demo

# 2. Создаем чистую пустую базу с тем же именем
createdb -h localhost -U $USER demo

# 3. Накатываем наш сохраненный бэкап
pg_restore -h localhost -U $USER -d demo backups/demo_backup.dump
```

## 2. Проектирование изменений структуры (Схема)

### Задача 1: Длина ВПП для аэропорта и самолета
**Изменения:**
- Добавить в таблицу `airports_data` столбец `max_runway_length INT`.
- Добавить в таблицу `airplanes_data` столбец `min_runway_length INT`.
- Обновить представления (views) `airports` и `airplanes`, чтобы эти новые поля отображались при обычных запросах.

**Обоснование:** 
Информация об аэропорте и о самолете — это их статические физические характеристики. Самое логичное место для хранения — это их базовые словари (таблицы `airports_data` и `airplanes_data`). Позже это позволит добавить проверку (Constraint/Trigger), может ли конкретный самолет приземлиться в конкретном аэропорту (`min_runway_length <= max_runway_length`).

### Задача 2: Блокировка мест на рейс
**Изменения:**
- Создать новую таблицу `flight_blocked_seats`.
  - `flight_id INT` (внешний ключ на `flights(flight_id)`).
  - `seat_no VARCHAR(4)` (номер места).
  - `reason TEXT` (причина блокировки — например, "сломано кресло").
  - Первичный ключ `PRIMARY KEY (flight_id, seat_no)`.

**Обоснование:**
Место блокируется не навсегда в самолете (тогда это было бы в таблице `seats`), а только на **конкретный рейс** (flight). Добавление флага в таблицу `boarding_passes` или `tickets` не подходит, так как они создаются под пассажиров (а заблокированное место означает отсутствие билета). Поэтому оптимальным решением является отдельная связующая таблица блокировок для конкретного рейса.

### Задача 3: Посадка рейса в резервном аэропорту
**Изменения:**
- Добавить в таблицу `flights` столбец `actual_arrival_airport CHAR(3)` (внешний ключ на `airports_data`).
- Добавить в таблицу `flights` столбец `divert_reason TEXT` (описание причины изменения порта).
- Обновить представление `timetable` (расписание), добавив эти столбцы, чтобы при выводе мы видели, куда рейс прилетел по факту.

**Обоснование:**
Таблица `flights` описывает фактический перелет. В ней уже есть `status` и фактическое время (`actual_arrival`). Поле маршрута (`route_no`) хранит запланированную информацию, которую мы не трогаем. А вот фактический аэропорт прибытия — это чистое свойство перелета, поэтому дополнительные колонки в таблице `flights` идеально ложатся в текущую архитектуру без потерь данных.
