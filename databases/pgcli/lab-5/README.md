## Теория: Хранимые функции и процедуры в PostgreSQL

### FUNCTION vs PROCEDURE
| | `CREATE FUNCTION` | `CREATE PROCEDURE` |
|---|---|---|
| Возвращает значение | Да (`RETURNS тип`) | Нет (только `OUT`-параметры) |
| Вызов | `SELECT f(...)` или в выражениях | `CALL p(...)` |
| Управление транзакциями | Нет (работает в рамке вызова) | Да (`COMMIT`/`ROLLBACK` внутри) |
| Когда использовать | Вычисления, проверки, скалярные результаты | Сложные операции с несколькими транзакциями |

В этой лабе используем **функции** (`RETURNS TEXT`) — они возвращают book_ref и вызываются через `SELECT`.

### PL/pgSQL — основы
```sql
CREATE OR REPLACE FUNCTION my_func(p_id INT)
RETURNS TEXT
LANGUAGE plpgsql
AS $$
DECLARE
    v_result TEXT;  -- переменная
BEGIN
    SELECT col INTO v_result FROM t WHERE id = p_id;

    IF NOT FOUND THEN
        RAISE EXCEPTION 'Запись % не найдена', p_id;  -- бросить ошибку
    END IF;

    RETURN v_result;
EXCEPTION
    WHEN OTHERS THEN           -- поймать любую ошибку
        RAISE NOTICE 'Ошибка: %', SQLERRM;
        RETURN NULL;
END;
$$;
```

### Последовательности (SEQUENCE)
```sql
CREATE SEQUENCE my_seq START WITH 1000 INCREMENT BY 1;
SELECT nextval('my_seq');  -- получить следующее значение
SELECT currval('my_seq');  -- текущее значение в сессии
```
Последовательности гарантируют уникальность даже при параллельных вставках (без блокировок).

### Атомарность транзакций
Функция выполняется внутри транзакции вызывающего кода. Если внутри функции произошло исключение (или `RAISE EXCEPTION`), **все изменения в функции откатываются автоматически**.

## Реализованные процедуры

### `check_free_seats(p_flight_id INT) → INT`
Вспомогательная функция. Возвращает количество свободных мест на рейсе:

```
(кол-во мест в самолёте по данному маршруту) − (кол-во уже проданных сегментов)
```

В базе `demo` физические места самолёта хранятся в таблице `seats`, привязаны к коду самолёта (`airplane_code`). Связь: `flights → routes → seats`.

### `book_ticket(passenger_id, passenger_name, flight_id, fare_conditions, price) → TEXT`
Основная процедура бронирования. Выполняет:
1. Проверку допустимого класса обслуживания (`Economy`, `Comfort`, `Business`).
2. Проверку существования рейса.
3. Проверку статуса рейса — бронирование разрешено только при статусе `Scheduled` или `On Time`.
4. **Проверку наличия свободных мест** через `check_free_seats`.
5. Генерацию уникального 6-символьного кода бронирования (`book_ref`).
6. Генерацию 13-значного номера билета (`ticket_no`) через последовательность `ticket_no_seq`.
7. Атомарную вставку в три таблицы: `bookings`, `tickets`, `segments`.

Возвращает `book_ref` (например: `G0NWW5`).

### Вспомогательные объекты
- **Последовательность `ticket_no_seq`**: генерирует уникальные номера билетов, начиная с 5 435 000 000 (выше максимального существующего в базе).
- **Функция `gen_book_ref()`**: генерирует случайный 6-символьный код из букв `A-Z` и цифр `0-9`.

## Примечания по схеме

В базе `demo` место назначается пассажиру не при бронировании, а при **регистрации** (таблица `boarding_passes`). Поэтому проверка «свободных мест» на этапе бронирования — это проверка на наполненность рейса по числу проданных сегментов, а не по занятым физическим местам.
