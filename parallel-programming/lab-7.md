## 1. Вступление

> В работе реализована база данных и программный модуль на C#, который сравнивает стандартный SQL JOIN и собственный алгоритм merge-join, аналогичный алгоритмам из файлов NSSFMC… и NonStrCMrg.cs.  
> Задача – сгенерировать две таблицы с неиндексированным символьным ключом, отсортировать их по ключу и выполнить объединение двумя способами: штатным JOIN в СУБД и реализованным мной merge-алгоритмом, в том числе с параллельной обработкой.

### 2.2. Пункт 2 – генерация данных как в NSSFMCSnpAi.cs

**Класс:** `ABGenerator`

**Логика:**

- Используется **алфавит из двух букв** `A` и `B`, как в NSSFMCSnpAi, длина ключа `L = 10`.
    
- Сначала генерируется нужное число **различных ключей** (например, 1000):
    
    ```csharp
    char[] buf = new char[KeyLength];
    for (int i = 0; i < buf.Length; i++)
        buf[i] = rand.Next(2) == 0 ? 'A' : 'B';
    ```
    
- Для каждого ключа:
    
    - создаётся **от 2 до 5 строк в таблице A** с этим ключом и случайными значениями `B, D1, D2`;
        
    - и **от 2 до 5 строк в таблице B** с этим ключом и случайными `C, E1, E2`.
        

> Таким образом, выполняется требование: _«классы эквивалентности содержали более одной записи»_ – для каждого ключа A есть несколько строк в обеих таблицах.

- Используется `DataTable` + `SqlBulkCopy`, как в NSSFMCSnpAi:
    
    ```csharp
    SqlBulkCopy bulkA = new SqlBulkCopy(con);
    bulkA.DestinationTableName = "dbo.A";
    bulkA.WriteToServer(wA);
    ```
    

- Открываешь `ABGenerator.cs`, кратко показываешь генерацию ключа и использование `SqlBulkCopy`.
    
- В Server Explorer открываешь `A` и `B` → «Показать данные» – видно, что:
    
    - ключи вида `AAAAABABBB`,
        
    - один и тот же ключ встречается несколько раз.

### 2.3. Сортировка и создание Asrt / Bsrt

**Класс:** `TableSorter`

```csharp
SELECT * INTO Asrt FROM A ORDER BY A;
SELECT * INTO Bsrt FROM B ORDER BY A;
```

- После генерации таблицы A и B **не отсортированы**.
    
- Для merge-join требуется, чтобы оба входных набора были **отсортированы по ключу**.
    
- Поэтому делаю отдельные копии `Asrt` и `Bsrt`, содержащие те же данные, но уже в порядке возрастания поля `A`.
    

- В Server Explorer → таблица `Asrt`, «Показать данные» – ключи идут по порядку; `A` в исходной таблице может быть вперемешку.

## 3. Merge-join по образцу NonStrCMrg.cs

### 3.1. Общая идея

**Класс:** `MergeAB`

Структура черпака:

```csharp
struct Bucket
{
    public string Key;      // значение A
    public List<int> Bs;    // все B для данного ключа в Asrt
}
```

**Алгоритм (говоришь на защите):**

1. Считаем, что **таблица A – ведущая**, `Asrt`, таблица B – ведомая, `Bsrt` (как в NonStrCMrg: StudS – ведущая, PBS – ведомая).
    
2. Открываем два `SqlDataReader`:
    
    ```csharp
    SELECT A, B FROM Asrt ORDER BY A;
    SELECT A, C FROM Bsrt ORDER BY A;
    ```
    
3. Функция `Scoop()` – загружает **черпак**:
    
    - читает текущую строку из `Asrt`,
        
    - запоминает её ключ в `bucket.Key`,
        
    - собирает в список `bucket.Bs` все значения `B` с этим же ключом (пока ключ не поменяется).
        
4. Далее `Merge()` идёт по `Bsrt`:
    
    - сравнивает `bucket.Key` и `keyB` (ключ текущей строки Bsrt):
        
        - если `bucket.Key < keyB` – ведущая таблица «отстаёт», делаем новый `Scoop()` (новый черпак);
            
        - если `bucket.Key > keyB` – ведомая «отстаёт», читаем следующую строку `Bsrt`;
            
        - если ключи равны – для текущего `C` обрабатываем **все элементы черпака** (все `B` для этого ключа).
            
5. Обработка совпадения:
    
    - считаем произведения `B * C` и суммируем по ключу:
        
        - так реализуем агрегат `SUM(A.B*B.C)` как в запросе `QAjB.sql`.
            
    - когда ключ меняется – записываем строку в `AjB_merge (A, sBC)`.
        

> По сути это классический merge-join, сложность O(n + m): мы каждый поток читания идём вперёд максимум один раз.

- В `MergeAB.cs` показываешь:
    
    - `Bucket` и `Scoop()`,
        
    - кусок `MergeInternal()` с логикой сравнения ключей и обработки совпадения.

## 4. Параллельная обработка черпака (пункт про «совместную обработку черпака и ведомой записи»)

**В том же `MergeAB`** у меня два режима:

- `MergeSequential()` – последовательный;
    
- `MergeParallel()` – параллельный.
    

Различие только в обработке черпака:

```csharp
if (parallel)
{
    object lockObj = new object();
    Parallel.ForEach(bucket.Bs, bVal =>
    {
        double prod = bVal * cVal;
        lock (lockObj)
        {
            localSum += prod;
        }
    });
}
else
{
    foreach (int bVal in bucket.Bs)
        localSum += bVal * cVal;
}
```


- Задача – **параллелизовать совместную обработку черпака и записи ведомой таблицы**.
    
- На уровне массивов это делается стандартно: `Parallel.For` / `Parallel.ForEach` по элементам массива.
    
- Здесь `bucket.Bs` – это массив значений `B` для текущего ключа; `cVal` – текущее `C` из ведомой таблицы.  
    Каждую пару `(B_i, C)` можно считать в отдельном потоке → мы это и делаем.
    
- Сумма `localSum` защищена `lock`, чтобы избежать гонок.
    

Можно честно сказать, что при таком размере черпака (2–5 элементов) **параллельная версия оказывается медленнее** из-за накладных расходов, что и видно по эксперименту.

## 5. Стандартный JOIN и сравнение времени

**Класс:** `Benchmark`

### 5.1. SQL JOIN (RunSqlJoin)

```csharp
SELECT A.A, SUM(A.B * B.C) AS sBC
INTO AjB_sql
FROM A INNER JOIN B ON A.A = B.A
GROUP BY A.A
ORDER BY A.A;
```

- Это прямой аналог `QAjB.sql` из методички.
    
- Результат в таблице `AjB_sql(A nchar(10), sBC int)`.
    

### 5.2. Замер времени

Каждый из трёх вариантов оборачивается в `Stopwatch`:

- `RunSqlJoin()` – стандартный SQL JOIN → `AjB_sql`.
    
- `RunMergeSequential()` – наш merge-join → `AjB_merge`.
    
- `RunMergeParallel()` – параллельный merge-join → `AjB_merge` (перезаписывает).
    

**Реальные результаты (можно озвучить):**

- SQL JOIN: ~146 мс
    
- Merge-Join (послед.): ~313 мс
    
- Merge-Join (паралл.): ~651 мс
    

**Как объяснять:**

- SQL JOIN выполняется целиком **внутри оптимизированного движка СУБД**, без гоняния данных через .NET.
    
- Наш merge-join читает данные `SqlDataReader`-ом, обрабатывает в C# и пишет обратно через `SqlBulkCopy`. Это отнимает время, поэтому он **медленнее**.
    
- Параллельный вариант в данном случае проигрывает последовательному, потому что:
    
    - размер черпака небольшой;
        
    - на каждый `Parallel.ForEach` есть накладные расходы планировщика + `lock`;
        
    - выигрыша от распараллеливания мало, а оверхед – значительный.

## 7. Возможные вопросы препода и короткие ответы

**1. Зачем делать сортировку?**  
Чтобы реализовать merge-join за O(n + m). Алгоритм предполагает, что обе последовательности отсортированы по ключу; тогда достаточно одного прохода по каждой без вложенных циклов.

**2. Почему ключ NCHAR и почему не индексирован?**  
Тип NCHAR(10) предписан ТЗ. Ключевое поле не объявляю ключом БД и не создаю индекс, чтобы не было скрытого автоматического индексирования – это требование ТЗ, чтобы алгоритмы сравнивались честно, без помощи индексов.

**3. Чем merge-join отличается от обычного JOIN?**  
JOIN в СУБД может выбрать один из трёх планов: nested-loop, hash join или merge join. Мы вручную реализуем merge-join: два отсортированных потока, один черпак, один проход. В нашем случае он полностью реализован на стороне клиента C#.

**4. Почему параллельная версия медленнее?**  
Мало работы на поток (черпаки маленькие), есть накладные расходы на создание/планирование потоков и блокировки. Поэтому параллелизм здесь не оправдывается – это тоже важный практический вывод.

**5. Как обеспечивается, что в классах эквивалентности более одной записи?**  
При генерации для каждого уникального ключа создаётся не меньше двух строк в A и не меньше двух в B (`minRowsPerKey = 2`). Это видно в `ABGenerator.GenerateData`.
