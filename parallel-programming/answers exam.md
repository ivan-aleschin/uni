
# Шпаргалка по билетам: Теория параллельного программирования

## 1. Классификация архитектур параллельных вычислительных систем

**Классификация Флинна** (по потокам команд и данных):

| Тип | Расшифровка | Описание | Пример |
|-----|-------------|----------|--------|
| **SISD** | Single Instruction Single Data | Обычный процессор, одна команда — одно данное | Старые CPU |
| **SIMD** | Single Instruction Multiple Data | Одна команда — много данных (векторные операции) | SSE, AVX |
| **MISD** | Multiple Instruction Single Data | Много команд — одно данное (редко встречается) | Отказоустойчивые системы |
| **MIMD** | Multiple Instruction Multiple Data | Много команд — много данных (независимые процессоры) | Многоядерные CPU |

**По организации памяти:**
- **Общая память (Shared Memory)**: все процессоры видят одну память → SMP, NUMA
- **Распределённая память (Distributed Memory)**: у каждого процессора своя память → кластеры, MPP

## 2. Алгоритм параллельной сортировки массива

### Чёт-нечёт сортировка (Odd-Even Sort)

**Идея**: разделить сравнения на чётные и нечётные фазы.

```

Фаза 0 (чётная): сравниваем пары (0,1), (2,3), (4,5) — параллельно Фаза 1 (нечётная): сравниваем пары (1,2), (3,4), (5,6) — параллельно Повторяем N раз

```

**Почему параллельна?** Пары не пересекаются → можно обрабатывать одновременно.

**Сложность**: O(N²) сравнений, но на N процессорах → O(N) времени.

### Быстрая сортировка (Parallel QuickSort)

**Идея**: разделить массив на части, сортировать части параллельно.

```

1. Выбрать pivot
2. Разделить массив на: меньше pivot | равно | больше pivot
3. Параллельно отсортировать левую и правую части

```

**Проблема**: неравномерное разбиение → плохая балансировка.

## 3. Параллельная архитектура SIMD

**SIMD** = Single Instruction Multiple Data — одна команда обрабатывает много данных.

**Принцип работы**:
```

Обычно: SIMD: A[0] + B[0] [A[0], A[1], A[2], A[3]] + [B[0], B[1], B[2], B[3]] A[1] + B[1] ↓ A[2] + B[2] Одна инструкция ADDPS A[3] + B[3]

```

**Примеры**: SSE (128 бит, 4 float), AVX2 (256 бит, 8 float), AVX-512 (512 бит, 16 float).

**Плюсы**: высокая производительность на однородных данных.
**Минусы**: плохо работает с ветвлениями (if/else).

## 4. Закон Амдала

**Формула**:
```

Ускорение = 1 / (S + P/N)

S = доля последовательного кода (нельзя распараллелить) P = доля параллельного кода (S + P = 1) N = количество процессоров

```

**Смысл**: даже при бесконечном числе процессоров ускорение ограничено последовательной частью.

**Пример**:
```

S = 0.1 (10% последовательно) P = 0.9 (90% параллельно) N = 4 процессора

Ускорение = 1 / (0.1 + 0.9/4) = 1 / 0.325 ≈ 3.08×

При N → ∞: Ускорение_max = 1 / 0.1 = 10×

```

**Вывод**: чтобы получить большое ускорение, нужно минимизировать последовательную часть.

## 5. Параллельная архитектура MISD

**MISD** = Multiple Instruction Single Data — много команд обрабатывают одно данное.

**Где используется**: почти нигде в классическом виде.

**Примеры применения**:
- **Отказоустойчивые системы**: несколько процессоров выполняют одни и те же вычисления на одних данных, результаты сравниваются → если различаются, есть сбой.
- **Конвейерная обработка**: данное проходит через несколько этапов обработки (но это спорно, больше похоже на pipeline).

**Практическое значение**: минимальное, в основном теоретический интерес.

## 6. Алгоритм Кэннона умножения матриц

**Цель**: распределить умножение матриц N×N между P процессорами (P = √P × √P сетка).

**Шаги**:
1. **Разбиение**: матрицы A и B делятся на блоки размером (N/√P) × (N/√P)
2. **Начальный сдвиг**:
   - Блоки A в строке i сдвигаются влево на i позиций
   - Блоки B в столбце j сдвигаются вверх на j позиций
3. **Основной цикл** (√P итераций):
   - Умножение соответствующих блоков
   - Циклический сдвиг: A влево, B вверх
4. **Сборка результата**

**Пример** (2×2 сетка):
```

A = [A00 A01] B = [B00 B01] [A10 A11] [B10 B11]

Процессор (0,0): A00 × B00 + A01 × B10 Процессор (0,1): A00 × B01 + A01 × B11 Процессор (1,0): A10 × B00 + A11 × B10 Процессор (1,1): A10 × B01 + A11 × B11

```

**Почему эффективно**: минимизация передачи данных, хорошее использование кэша.

## 7. Параллельная архитектура MIMD

**MIMD** = Multiple Instruction Multiple Data — много команд обрабатывают много данных.

**Суть**: несколько независимых процессоров, каждый выполняет свою программу на своих данных.

**Типы**:
- **Общая память (Shared Memory)**: процессоры видят одну память → SMP, NUMA
- **Распределённая память (Distributed Memory)**: каждый процессор имеет свою память → кластеры

**Примеры**: 
- Многоядерные процессоры (Intel i7, Ryzen)
- Суперкомпьютеры
- Кластеры серверов

**Плюсы**: максимальная гибкость, каждый процессор работает независимо.
**Минусы**: сложность синхронизации, проблемы с обменом данными.

## 8. Параллельная реализация алгоритма транзитивного замыкания

**Задача**: для графа найти матрицу достижимости (reach[i][j] = true, если есть путь i→j).

**Алгоритм Флойда-Уоршелла** (для булевых матриц):
```

for k = 0 to N-1: for i = 0 to N-1 (параллельно): for j = 0 to N-1: reach[i][j] = reach[i][j] OR (reach[i][k] AND reach[k][j])

```

**Параллелизм**: внутренний цикл по `i` распараллеливается (каждый поток обрабатывает свои строки).

**Конвейерная реализация**:
```

Этап 1: k=0 → обработка строк 0..N-1 Этап 2: k=1 → обработка строк 0..N-1 ... Этап N: k=N-1 → обработка строк 0..N-1

```

Каждый этап зависит от предыдущего → **конвейер** (pipeline).

## 9. SMP архитектура

**SMP** = Symmetric MultiProcessing — симметричная многопроцессорная обработка.

**Характеристики**:
- Все процессоры **равноправны**
- **Общая память** (единое адресное пространство)
- **Общая шина** для доступа к памяти
- **Единая ОС** управляет всеми процессорами

**Схема**:
```

[CPU 0] [CPU 1] [CPU 2] [CPU 3] \ | | / \ | | / [ Общая шина ] | [ Общая RAM ]

````

**Плюсы**: простота программирования (одна память).
**Минусы**: ограниченная масштабируемость (шина становится узким местом при > 8-16 CPU).

**Примеры**: обычные многоядерные процессоры (Intel, AMD).

## 10. Объект Thread

**Thread** — объект для создания и управления потоком выполнения.

**Основные методы** (C#):
```csharp
Thread t = new Thread(DoWork);  // Создание
t.Start();                       // Запуск
t.Join();                        // Ожидание завершения
t.Abort();                       // Принудительная остановка (устарело)
Thread.Sleep(100);               // Приостановка на 100 мс
````

**Свойства**:

- `t.IsAlive` — работает ли поток
- `t.Priority` — приоритет (Low, Normal, High)
- `t.Name` — имя потока (для отладки)

**Пример**:

```csharp
void DoWork() {
    Console.WriteLine("Поток работает");
}

Thread t = new Thread(DoWork);
t.Start();
t.Join();  // Ждём завершения
```

## 11. MPP архитектура

**MPP** = Massively Parallel Processing — массивно-параллельная обработка.

**Характеристики**:

- Много процессоров (**сотни/тысячи**)
- **Распределённая память** (каждый процессор имеет свою)
- **Высокоскоростная сеть** для связи (Infiniband, custom networks)
- **Специализированная ОС** или middleware

**Схема**:

```
[CPU+RAM] — [CPU+RAM] — [CPU+RAM]
    |            |            |
  [Коммутатор / высокоскоростная сеть]
    |            |            |
[CPU+RAM] — [CPU+RAM] — [CPU+RAM]
```

**Отличие от кластера**: MPP — единая система с специальной архитектурой, кластер — набор стандартных серверов.

**Примеры**: суперкомпьютеры (Cray, IBM Blue Gene).

**Плюсы**: отличная масштабируемость. **Минусы**: сложность программирования, высокая стоимость.

## 12. Объект ManualResetEvent

**ManualResetEvent** — примитив синхронизации для уведомления между потоками.

**Состояния**:

- **Signaled** (сигнализировано): потоки проходят
- **Non-signaled** (не сигнализировано): потоки блокируются

**Методы**:

```csharp
ManualResetEvent mre = new ManualResetEvent(false);  // Начальное состояние: non-signaled

mre.WaitOne();    // Блокирует поток, пока не станет signaled
mre.Set();        // Переводит в signaled (все ждущие потоки просыпаются)
mre.Reset();      // Переводит в non-signaled
```

**Пример использования**:

```csharp
ManualResetEvent ready = new ManualResetEvent(false);

// Поток 1: ждёт сигнала
Thread t1 = new Thread(() => {
    Console.WriteLine("Жду сигнала...");
    ready.WaitOne();  // Блокируется здесь
    Console.WriteLine("Получил сигнал!");
});

// Главный поток: даёт сигнал
t1.Start();
Thread.Sleep(1000);
ready.Set();  // Разблокирует поток 1
```

**Отличие от AutoResetEvent**: ManualResetEvent не сбрасывается автоматически (все ждущие потоки проходят).

## 13. Оценка эффективности параллельных вычислений

**Основные метрики**:

### 1. Ускорение (Speedup)

```
S = T_seq / T_par

T_seq = время последовательного выполнения
T_par = время параллельного выполнения
```

**Идеал**: S = N (линейное ускорение на N процессорах) **Реальность**: S < N из-за overhead

### 2. Эффективность (Efficiency)

```
E = S / N × 100%

N = количество процессоров
```

**Идеал**: E = 100% **Хорошо**: E > 70% **Плохо**: E < 50%

### 3. Масштабируемость

Как меняется ускорение при увеличении числа процессоров?

- **Сильная масштабируемость**: фиксированная задача, растёт N → растёт S
- **Слабая масштабируемость**: задача растёт пропорционально N → время постоянно

**Пример**:

```
N=1: T=100мс, S=1.0, E=100%
N=4: T=30мс, S=3.3, E=83%
N=8: T=20мс, S=5.0, E=63%
```

Вывод: эффективность падает с ростом N (накладные расходы).

## 14. Работа с потоками в C#

### Thread (явное управление)

```csharp
Thread t = new Thread(() => {
    Console.WriteLine("Поток работает");
});
t.Start();
t.Join();  // Ждём завершения
```

### Task (современный подход)

```csharp
Task task = Task.Run(() => {
    Console.WriteLine("Task работает");
});
task.Wait();  // Ждём завершения
```

### Parallel.For (автоматическое распределение)

```csharp
Parallel.For(0, 100, i => {
    Console.WriteLine($"Обработка {i}");
});
```

### async/await (асинхронность)

```csharp
async Task DoWorkAsync() {
    await Task.Delay(1000);
    Console.WriteLine("Готово");
}
```

**Когда что использовать**:

- **Thread**: низкоуровневое управление, редко нужен
- **Task**: стандартный выбор для параллелизма
- **Parallel**: для параллельных циклов
- **async/await**: для I/O операций (не CPU-bound задачи)

## 15. Выбор параллельного алгоритма

**Критерии выбора**:

### 1. Тип задачи

- **Data-parallel** (однородные операции над данными) → SIMD, Parallel.For
- **Task-parallel** (независимые задачи) → Task, Thread Pool
- **Pipeline** (последовательные этапы) → конвейерная обработка

### 2. Размер данных

- **Малый** (< 1000 элементов) → возможно, последовательно лучше (overhead)
- **Средний** (1000-100000) → параллелизм эффективен
- **Большой** (> 100000) → обязательно параллелить

### 3. Архитектура системы

- **SMP** (общая память) → потоки, Parallel
- **MPP/кластер** (распределённая память) → MPI, распределённые алгоритмы

### 4. Зависимости между данными

- **Независимые** → легко параллелить (embarrassingly parallel)
- **Слабые зависимости** → можно параллелить с синхронизацией
- **Сильные зависимости** → сложно/невозможно параллелить

**Пример**: умножение матриц

- Малые (< 100×100): последовательно
- Средние (100-1000): блочное + потоки
- Большие (> 1000): Cannon/Strassen + MPP

## 16. Построение исполняемой программы для параллельной системы

**Этапы**:

### 1. Анализ алгоритма

- Выделить параллельные участки
- Определить зависимости
- Оценить потенциальное ускорение (закон Амдала)

### 2. Декомпозиция

- **По данным**: разбить данные на части (data parallelism)
- **По задачам**: разбить на независимые задачи (task parallelism)

### 3. Выбор технологии

- **Общая память**: OpenMP, Pthreads, C# Task/Thread
- **Распределённая память**: MPI
- **Гибридная**: MPI + OpenMP

### 4. Реализация

```csharp
// Пример: параллельная обработка массива
int[] data = new int[1000000];

Parallel.For(0, data.Length, i => {
    data[i] = Process(data[i]);  // Независимая обработка каждого элемента
});
```

### 5. Оптимизация

- Минимизировать синхронизацию
- Балансировка нагрузки (равномерное распределение работы)
- Локальность данных (cache-friendly алгоритмы)

### 6. Тестирование

- Проверка корректности
- Измерение ускорения
- Профилирование (поиск узких мест)

## 17. Показатель эффекта распараллеливания

### Ускорение (Speedup)

```
S(N) = T(1) / T(N)

T(1) = время на 1 процессоре
T(N) = время на N процессорах
```

**Типы ускорения**:

- **Линейное**: S(N) = N (идеал)
- **Сублинейное**: S(N) < N (обычно)
- **Суперлинейное**: S(N) > N (редко, из-за эффектов кэша)

### Эффективность

```
E(N) = S(N) / N = T(1) / (N × T(N))
```

**Интерпретация**:

- E = 1.0 (100%): идеальное использование
- E = 0.8 (80%): хорошо
- E = 0.5 (50%): посредственно
- E < 0.3 (30%): плохо

### Масштабируемость

```
Изомасштабируемость: T(N) = const при росте задачи пропорционально N
```

**Пример**:

```
Задача: сортировка 1М элементов
N=1: T=1000мс, S=1.0, E=100%
N=2: T=550мс, S=1.8, E=91%
N=4: T=300мс, S=3.3, E=83%
N=8: T=180мс, S=5.6, E=70%
```

## 18. Параллельное исполнение машинной программы

**Уровни параллелизма в процессоре**:

### 1. Конвейер (Pipeline)

```
Такт 1: [Fetch]
Такт 2: [Decode] [Fetch]
Такт 3: [Execute] [Decode] [Fetch]
Такт 4: [Memory] [Execute] [Decode] [Fetch]
```

Несколько инструкций выполняются одновременно на разных стадиях.

### 2. Суперскалярность

Процессор выполняет несколько инструкций **за один такт** (если они независимы).

```
ADD R1, R2, R3  } Выполняются
MUL R4, R5, R6  } одновременно (разные исполнительные блоки)
```

### 3. Out-of-Order Execution

Процессор переупорядочивает инструкции для максимальной загрузки.

```
Исходный порядок:     Реальный порядок:
1. LOAD R1, [addr]    1. LOAD R1, [addr]
2. ADD R2, R1, R3     3. MUL R4, R5, R6  (выполняется пока ждём LOAD)
3. MUL R4, R5, R6     2. ADD R2, R1, R3
```

### 4. SIMD

Одна инструкция обрабатывает несколько данных (SSE, AVX).

### 5. Многопоточность (SMT/Hyper-Threading)

Один физический процессор выполняет несколько потоков одновременно.

**Итог**: современный процессор параллелит на многих уровнях автоматически.

## 19. Распараллеливание на разных уровнях

### Уровень команд (Instruction-Level Parallelism)

```c
a = b + c;   // Инструкции 1-2
d = e * f;   // Инструкции 3-4
```

Процессор выполняет обе операции одновременно (если независимы).

### Уровень выражений

```c
x = a + b;
y = c * d;
z = x - y;  // Зависит от x и y
```

Первые два выражения независимы → параллельно.

### Уровень циклов

```c
for (int i = 0; i < N; i++) {
    C[i] = A[i] + B[i];  // Независимые итерации
}
```

Можно распараллелить через OpenMP/Parallel.For.

### Уровень функций/модулей

```c
Task t1 = Task.Run(() => ProcessFileA());
Task t2 = Task.Run(() => ProcessFileB());
Task.WaitAll(t1, t2);
```

Независимые функции выполняются параллельно.

### Уровень заданий (Job-Level)

```
Задание 1: Рендеринг кадра 1
Задание 2: Рендеринг кадра 2
Задание 3: Рендеринг кадра 3
```

Распределяются по узлам кластера.

**Чем выше уровень → тем больше параллелизма, но сложнее управление.**

## 20. Выбор оптимального алгоритма для системы

**Факторы**:

### 1. Архитектура системы

|Архитектура|Рекомендуемый алгоритм|
|---|---|
|SIMD|Векторные операции (SSE/AVX)|
|SMP|Потоки, Parallel.For|
|NUMA|Учёт локальности памяти|
|MPP/Cluster|Распределённые алгоритмы (MPI)|

### 2. Размер задачи

- **Малая**: последовательный (overhead > выигрыш)
- **Средняя**: параллельный на SMP
- **Большая**: распределённый на кластере

### 3. Коммуникационные затраты

```
T_total = T_comp + T_comm

Если T_comm > T_comp → параллелизм неэффективен
```

**Пример**: умножение матриц

- **100×100**: последовательно (overhead велик)
- **1000×1000**: блочный + потоки (SMP)
- **10000×10000**: Cannon на кластере (MPP)

### 4. Балансировка нагрузки

- **Статическое распределение**: для равномерных задач
- **Динамическое**: для неравномерных (work stealing)

## 21. Нахождение наилучшей топологии

**Топология** — схема соединения процессоров в системе.

### Основные топологии:

**1. Полносвязная (All-to-All)**

```
Каждый процессор связан со всеми остальными
Диаметр: 1
Стоимость: O(N²) связей
```

Плюсы: минимальная задержка. Минусы: дорого при большом N.

**2. Кольцо (Ring)**

```
P0 — P1 — P2 — P3 — P0
Диаметр: N/2
Стоимость: O(N) связей
```

Плюсы: дёшево. Минусы: большая задержка.

**3. Решётка (Mesh/Grid)**

```
P0 — P1 — P2
|    |    |
P3 — P4 — P5
|    |    |
P6 — P7 — P8

Диаметр: 2√N
```

Используется в GPU, NPU.

**4. Гиперкуб (Hypercube)**

```
Для 8 процессоров (3D куб):
Диаметр: log₂(N)
Связей на процессор: log₂(N)
```

Хороший компромисс между стоимостью и задержкой.

**5. Дерево (Tree)**

```
      P0
     /  \
   P1    P2
  / \    / \
P3  P4 P5  P6
```

Используется в иерархических задачах.

### Выбор топологии:

- **Алгоритм с локальными обменами** (умножение матриц) → Mesh
- **All-to-all коммуникации** → полносвязная или гиперкуб
- **Иерархическая задача** (древовидная редукция) → дерево

## 22. Семафоры как средство синхронизации

**Семафор** — счётчик с двумя атомарными операциями.

### Операции:

```
P (Wait, Down, захват):
  if (S > 0) S--;
  else блокировать поток

V (Signal, Up, освобождение):
  S++;
  разбудить один ждущий поток (если есть)
```

### Типы семафоров:

**Бинарный** (S ∈ {0, 1}):

```
S = 1  // Ресурс свободен
P(S)   // Захват (S = 0)
// Критическая секция
V(S)   // Освобождение (S = 1)
```

**Считающий** (S ∈ {0, 1, 2, ...}):

```
S = 3  // 3 свободных ресурса
P(S)   // S = 2 (занят один ресурс)
P(S)   // S = 1
P(S)   // S = 0 (все заняты)
P(S)   // Блокировка (ждём освобождения)
```

### Пример использования (C#):

```csharp
Semaphore sem = new Semaphore(1, 1);  // Начальное значение = 1, макс = 1

sem.WaitOne();  // P(S)
// Критическая секция
sem.Release();  // V(S)
```

**Применение**: ограничение доступа к ресурсу (например, пул из 5 потоков обработки).

## 23. NUMA архитектура

**NUMA** = Non-Uniform Memory Access — неоднородный доступ к памяти.

**Принцип**:

- Каждый процессор имеет **локальную память** (быстрый доступ)
- Может обращаться к **чужой памяти** (медленный доступ)

**Схема**:

```
[CPU 0 + RAM 0] ←→ Interconnect ←→ [CPU 1 + RAM 1]

Доступ CPU 0 к RAM 0: быстро (local)
Доступ CPU 0 к RAM 1: медленно (remote)
```

**Проблема**: если поток на CPU 0 работает с данными в RAM 1 → замедление.

**Решение**: привязка потоков к процессорам (thread affinity), размещение данных близко к обрабатывающему процессору.

**Отличие от SMP**:

- SMP: все процессоры видят память одинаково
- NUMA: время доступа зависит от расположения

**Примеры**: AMD EPYC, Intel Xeon (многосокетные системы).

## 24. Кластерная архитектура на ЛВС

**Кластер** — набор независимых компьютеров, связанных сетью и работающих как единая система.

**Компоненты**:

1. **Узлы** (nodes): обычные серверы/ПК
2. **Сеть**: Ethernet (1-100 Гбит/с) или Infiniband (> 100 Гбит/с)
3. **ПО**: MPI, OpenMPI, SLURM (планировщик заданий)

**Схема**:

```
[Узел 1] ——┐
[Узел 2] ——┼—— [Коммутатор] —— [Головной узел]
[Узел 3] ——┘
```

**Типы кластеров**:

- **Вычислительный** (HPC): для научных расчётов
- **Высокой доступности** (HA): для отказоустойчивости
- **Балансировки нагрузки**: для веб-серверов

**Программирование**:

```c
// MPI: отправка данных между узлами
MPI_Send(data, size, MPI_INT, dest_rank, tag, MPI_COMM_WORLD);
MPI_Recv(data, size, MPI_INT, src_rank, tag, MPI_COMM_WORLD, &status);
```

**Плюсы**: масштабируемость, дешевизна (стандартное железо). **Минусы**: высокая латентность сети (по сравнению с SMP).

## 25. Взаимодействие последовательных процессов

### 1. Разделяемая память (Shared Memory)

```csharp
int shared_var = 0;

// Поток 1
lock(lockObj) {

lock(lockObj) {
    shared_var--;
```

### 2. Передача сообщений (Message Passing)

### 3. Примитивы синхронизации

- **Semaphore**: ограничение доступа
- **Event**: уведомление о событии
- **Barrier**: ожидание всех потоков

### 4. Каналы (Channels)


**Выбор механизма**:

- **Сообщения**: распределённые системы, большие блоки данных

## 26. Системы с ассоциативным распределением ресурсов

**Ассоциативная память** (Content-Addressable Memory, CAM) — память, в которой поиск ведётся **по содержимому**, а не по адресу.

**Принцип**:

```
Обычная память:         Ассоциативная:
Адрес → Данные         Шаблон → Совпадающие данные

RAM[0x1000] = "cat"    Поиск "ca*" → ["cat", "car", "cap"]
```

**Применение в параллельных системах**:

- **Ассоциативные процессоры**: каждый элемент памяти имеет свою логику обработки
- **Поиск по шаблону**: все подходящие элементы обрабатываются **одновременно**

**Пример**:

```
Задача: найти все элементы > 100 в массиве

Обычно (последовательно):
for i in array:
  if i > 100: result.add(i)

Ассоциативно (параллельно):
Все элементы сравниваются с 100 одновременно
Отмечаются подходящие
```

**Плюсы**: сверхвысокий параллелизм для поисковых задач. **Минусы**: дорого, специализированное оборудование.

**Примеры**: сетевое оборудование (поиск в таблицах маршрутизации), базы данных (индексы).

## 28. Параллельное сложение матриц блочным методом

**Идея**: разбить матрицы на блоки, складывать блоки параллельно.

**Пример**: матрицы 1000×1000, 4 потока, блоки 500×500

```
A = [A00 A01]    B = [B00 B01]    C = [C00 C01]
    [A10 A11]        [B10 B11]        [C10 C11]

Поток 1: C00 = A00 + B00
Поток 2: C01 = A01 + B01
Поток 3: C10 = A10 + B10
Поток 4: C11 = A11 + B11
```

**Преимущества**:

1. **Кэш-эффективность**: блок помещается в L1/L2 кэш
2. **Балансировка**: каждый поток обрабатывает равный объём
3. **Масштабируемость**: легко увеличить число блоков/потоков

**Код**:

```csharp
int blockSize = N / numThreads;

Parallel.For(0, numThreads, t => {
    int startRow = t * blockSize;
    int endRow = (t == numThreads - 1) ? N : startRow + blockSize;
    
    for (int i = startRow; i < endRow; i++) {
        for (int j = 0; j < N; j++) {
            C[i, j] = A[i, j] + B[i, j];
        }
    }
});
```

**Сложность**: O(N²) операций, на P процессорах → O(N²/P) времени.
