# Теоретическая справка по технологиям параллельного программирования

## Лабораторная работа №1: Семафоры

### Что такое семафор?
Семафор — это механизм синхронизации для управления доступом к общему ресурсу в многопоточной/многопроцессной среде. Представляет собой счётчик с двумя атомарными операциями.

### Операции семафора
- **P (Wait, захват)**: уменьшает счётчик, если счётчик = 0, то блокирует поток до освобождения
- **V (Signal, освобождение)**: увеличивает счётчик, разблокирует ожидающие потоки

### Зачем нужен семафор?
Без семафора два процесса могут одновременно записывать в файл → **повреждение данных**.
С семафором: только один процесс работает с файлом, остальные ждут → **данные целостны**.

### Почему через файл (как в лабе)?
- Простота реализации (не требует API ОС)
- Работает между независимыми процессами
- Минус: медленнее, чем системные семафоры (но для учебных целей подходит)

## Лабораторная работа №2: SSE (XMM-регистры)

### Что такое SIMD?
**SIMD** (Single Instruction Multiple Data) — одна инструкция обрабатывает несколько данных одновременно.

### Что такое SSE?
**SSE** (Streaming SIMD Extensions) — набор инструкций процессора для векторных операций.

### Что такое XMM-регистры?
**XMM** — это 128-битные регистры процессора (XMM0-XMM15).
Один XMM-регистр вмещает:
- 4 × float (32 бита)
- 2 × double (64 бита)
- 16 × byte

### Пример работы
```
Обычное сложение (скаляр):
A[0] + B[0] = C[0]  ← 1 операция
A[1] + B[1] = C[1]  ← 1 операция
A[2] + B[2] = C[2]  ← 1 операция
A[3] + B[3] = C[3]  ← 1 операция
Итого: 4 операции

SSE (вектор):
[A[0], A[1], A[2], A[3]] + [B[0], B[1], B[2], B[3]] = [C[0], C[1], C[2], C[3]]
Итого: 1 операция (в 4 раза быстрее!)
```

### Почему быстрее?
- **Параллелизм на уровне данных**: процессор складывает 4 числа за одну инструкцию
- **Меньше итераций цикла**: вместо N итераций — N/4
- **Эффективность**: теоретически до 4× ускорения

### Почему не всегда 4×?
- Накладные расходы на загрузку/сохранение данных в регистры
- Ограничения памяти (bandwidth)
- Обработка "хвоста" (если размер не кратен 4)

## Лабораторная работа №3: AVX2

### Что такое AVX/AVX2?
**AVX** (Advanced Vector Extensions) — расширение SIMD, пришедшее после SSE.
**AVX2** — вторая версия с дополнительными инструкциями.

### Отличие от SSE
| Характеристика | SSE (XMM)    | AVX2 (YMM)   |
|---------------|--------------|--------------|
| Размер регистра | 128 бит    | 256 бит      |
| float за раз  | 4            | 8            |
| int за раз    | 4            | 8            |
| Теор. ускорение | до 4×      | до 8×        |

### Пример
```
SSE:  [A0, A1, A2, A3] + [B0, B1, B2, B3] = 4 элемента
AVX2: [A0, A1, A2, A3, A4, A5, A6, A7] + [...] = 8 элементов
```

### Почему быстрее SSE?
В 2 раза больше данных за одну операцию → меньше итераций цикла.

### Многопоточность + AVX2
В лабе 3 комбинируется:
- **AVX2**: 8 элементов за раз (SIMD)
- **Многопоточность**: несколько потоков работают параллельно

Результат: **SIMD × потоки** = до 8 × 6 = 48× теоретически (практически ~10-15×)

## Лабораторные работы №4-5: Многопоточность

### Что такое поток (Thread)?
**Поток** — это последовательность выполнения инструкций внутри процесса.
Один процесс может иметь несколько потоков, работающих параллельно.

### Параллелизм vs Конкурентность
- **Конкурентность**: потоки чередуются на одном ядре (иллюзия параллелизма)
- **Параллелизм**: потоки работают одновременно на разных ядрах (настоящий параллелизм)

### Task vs Parallel.For (C#)

**Task** — явное создание задачи:
```csharp
Task t1 = Task.Run(() => DoWork(1));
Task t2 = Task.Run(() => DoWork(2));
Task.WaitAll(t1, t2);
```
Контроль: полный (сами распределяем работу)

**Parallel.For** — автоматическое распределение:
```csharp
Parallel.For(0, N, i => DoWork(i));
```
Контроль: минимальный (фреймворк сам распределяет)

### Горизонтальное распределение
**Пример**: матрица 1000×1000, 4 потока
```
Поток 1: строки [0..249]
Поток 2: строки [250..499]
Поток 3: строки [500..749]
Поток 4: строки [750..999]
```

Каждый поток работает со своей областью памяти → **нет конфликтов** → быстро.

### Почему не всегда N× ускорение на N потоках?

**Закон Амдала**:
```
Ускорение = 1 / (S + P/N)

S = доля последовательного кода (нельзя распараллелить)
P = доля параллельного кода
N = количество потоков
```

**Пример**: 10% кода последовательно, 90% параллельно, 4 потока
```
Ускорение = 1 / (0.1 + 0.9/4) = 1 / 0.325 ≈ 3.08×
```

Максимум при бесконечных потоках: 1/0.1 = 10×

**Причины неидеального ускорения**:
1. **Overhead** (накладные расходы): создание/синхронизация потоков
2. **Неравномерность работы**: один поток закончил раньше, ждёт других
3. **Конкуренция за память**: потоки борются за доступ к RAM/кэшу
4. **Синхронизация**: блокировки, барьеры замедляют выполнение

## Лабораторная работа №5: Специализированные алгоритмы

### Чёт-нечёт сортировка (Odd-Even Sort)

**Суть**: на чётных фазах сравниваем пары (0,1), (2,3), (4,5)...
На нечётных: (1,2), (3,4), (5,6)...

**Почему параллельна?**
Пары не пересекаются → можно обрабатывать **одновременно**.
```
Фаза 0 (чётная): [5,2] [8,1] [9,3] ← 3 потока работают параллельно
Фаза 1 (нечётная): [5,1] [8,3] ← 2 потока работают параллельно
```

### Метод Кэннона (умножение матриц)

**Идея**: разбить матрицы на блоки, перемножить блоки с циклическими сдвигами.

**Зачем?**
- Обычное умножение: O(N³) операций
- Блочное: лучше использует кэш процессора → быстрее

**Циклический сдвиг**:
```
Шаг 1: блоки A сдвигаются влево, блоки B — вверх
Шаг 2: умножаются соответствующие блоки
Шаг 3: повторяется √P раз
```

**Почему быстрее?**
- Блоки помещаются в L1/L2 кэш → меньше обращений к RAM
- Параллелизм: каждый процесс обрабатывает свой блок

### Ленточный метод

**Идея**: разбить матрицу A на горизонтальные полосы (ленты).
```
Матрица A (1000×1000), 4 потока:
Поток 1: строки [0..249] × вся матрица B
Поток 2: строки [250..499] × вся матрица B
Поток 3: строки [500..749] × вся матрица B
Поток 4: строки [750..999] × вся матрица B
```

**Плюсы**: простота реализации
**Минусы**: каждый поток читает всю матрицу B (дублирование данных)

## Лабораторная работа №6: Абстрактная матричная машина

### Что такое АММ?

**Абстрактная матричная машина** — универсальная структура для операций над матрицами любого типа.

**Суть**: вместо конкретных операций (+, ×) используем абстрактные (Add, Mult).

### Зачем это нужно?

**Пример**: алгоритм Флойда-Уоршелла

Для кратчайших путей:
```
dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])
           ↓
         "сложение" = min
         "умножение" = +
```

Для связности графа (булевы матрицы):
```
reach[i][j] = reach[i][j] OR (reach[i][k] AND reach[k][j])
            ↓
          "сложение" = OR
          "умножение" = AND
```

**Один алгоритм** — разные операции → универсальность!

### Алгоритм Флойда-Уоршелла

**Задача**: найти кратчайшие пути между всеми парами вершин.

**Алгоритм**:
```
Для каждой промежуточной вершины k:
  Для каждой пары (i, j):
    Если путь i→k→j короче текущего i→j:
      Обновить расстояние
```

**Сложность**: O(N³) — медленно, но параллелится по парам (i,j).

**Восстановление пути**: дополнительная матрица `next[i][j]` хранит следующую вершину на пути i→j.

## Лабораторная работа №7: Merge-Join

### Что такое Join?

**Join** — операция объединения двух таблиц по общему ключу.
```
Таблица A:        Таблица B:        Результат (A JOIN B):
K | B             K | C             K | B | C
1 | 10            1 | 5             1 | 10 | 5
1 | 20            1 | 7             1 | 10 | 7
2 | 15            3 | 8             1 | 20 | 5
                                    1 | 20 | 7
```

### Что такое Merge-Join?

**Merge-Join** — алгоритм объединения для **отсортированных** таблиц.

**Принцип**: два указателя движутся по отсортированным таблицам, при совпадении ключей — объединяем.
```
Asrt: [1,1,2,3,5] ← указатель A
Bsrt: [1,1,3,4,6] ← указатель B

K=1: совпало → записываем результат
K=2: только в A → пропускаем
K=3: совпало → записываем результат
```

### Почему быстрее обычного Join?

**Обычный Join (Nested Loop)**: O(N × M) — для каждой строки A ищем в B
**Merge-Join**: O(N + M) — **один проход** по каждой таблице

### Параллельный Merge-Join

**Идея**: разбить диапазон ключей между потоками.
```
Ключи 1..1000, 4 потока:
Поток 1: ключи [1..250]
Поток 2: ключи [251..500]
Поток 3: ключи [501..750]
Поток 4: ключи [751..1000]
```

Каждый поток:
1. Читает свой диапазон из A и B
2. Делает Merge-Join локально
3. Записывает результат

**Зачем?**
- Меньше данных на поток → помещается в кэш
- Потоки не конфликтуют (разные диапазоны)
- Ускорение в N раз (N = кол-во потоков)

## Общие концепции

### Чанк (Chunk)

**Чанк** — порция данных для обработки одним потоком.

**Пример**: массив из 1000 элементов, 4 потока
```
Чанк 1: элементы [0..249]   ← Поток 1
Чанк 2: элементы [250..499] ← Поток 2
Чанк 3: элементы [500..749] ← Поток 3
Чанк 4: элементы [750..999] ← Поток 4
```

**Зачем чанки?**
- Балансировка нагрузки (каждый поток получает примерно одинаково работы)
- Уменьшение синхронизации (потоки работают независимо)

### Динамическое vs Статическое распределение

**Статическое**: заранее делим работу поровну
```
4 потока → каждому по 250 элементов
```
Минус: если один поток закончил раньше, он простаивает.

**Динамическое**: потоки берут чанки по мере освобождения
```
Поток освободился → берёт следующий свободный чанк
```
Плюс: лучшая балансировка, нет простоя.

### Ускорение (Speedup)
```
Speedup = T_последовательно / T_параллельно
```

**Пример**:
- Последовательно: 100 мс
- Параллельно (4 потока): 30 мс
- Speedup = 100/30 = 3.33×

### Эффективность (Efficiency)
```
Efficiency = Speedup / N_потоков × 100%
```

**Пример**: Speedup = 3.33×, потоков = 4
```
Efficiency = 3.33/4 = 83%
```

**Идеал**: 100% (линейное ускорение)
**Реальность**: 60-85% (хорошо), < 50% (плохо)

### Почему эффективность не 100%?

1. **Overhead**: создание потоков, синхронизация
2. **Неравномерность**: один поток работает дольше других
3. **Память**: потоки конкурируют за доступ к RAM
4. **Последовательные участки**: часть кода нельзя распараллелить

## Итоговая таблица технологий

| Технология | Что ускоряет | Как работает | Теор. ускорение |
|-----------|-------------|--------------|-----------------|
| **SSE (XMM)** | Операции над массивами | 4 элемента за раз | до 4× |
| **AVX2 (YMM)** | Операции над массивами | 8 элементов за раз | до 8× |
| **Многопоточность** | Независимые задачи | N ядер работают параллельно | до N× |
| **SSE + потоки** | Матричные операции | 4 элемента × N потоков | до 4N× |
| **AVX2 + потоки** | Матричные операции | 8 элементов × N потоков | до 8N× |
| **Merge-Join** | JOIN в БД | Один проход по данным | O(N+M) vs O(N×M) |
| **Блочные алгоритмы** | Умножение матриц | Кэш-эффективность | 2-5× |

## Практические выводы

### Когда использовать SIMD (SSE/AVX)?
✅ Большие массивы (> 1000 элементов)
✅ Простые операции (+, -, ×, /)
❌ Малые массивы (< 100) — overhead больше выигрыша
❌ Сложная логика (if/else) — плохо векторизуется

### Когда использовать многопоточность?
✅ Независимые задачи
✅ Большой объём данных
✅ Много ядер на процессоре
❌ Малый объём — overhead на создание потоков
❌ Много синхронизации — блокировки убивают производительность

### Оптимальное количество потоков?
```
N_потоков ≈ N_ядер (или N_ядер - 1)
```
Больше потоков чем ядер → **конкуренция** → медленнее.

### Комбинирование технологий
**Лучший результат**: SIMD + многопоточность
- SIMD: ускорение на уровне данных (4-8×)
- Потоки: ускорение на уровне задач (2-6×)
- Итого: 10-30× на практике

**Пример**: умножение матриц 1024×1024
- Скаляр: 100 мс
- SSE: 30 мс (3.3×)
- SSE + 4 потока: 10 мс (10×)
