## Задание 2: Зомби-процессы

### Теория:
- **Зомби-процесс** - завершенный процесс, чей статус не прочитан родителем
- **Статус Z** в ps - процесс в состоянии зомби
- **Причина** - родитель не вызвал wait() после завершения потомка
- **Решение** - вызов wait() или завершение родителя

### Демонстрация:
```bash
# Запуск программы zombie в фоне
echo "=== Запуск демонстрации зомби-процессов ==="
./zombie &

# Запоминаем PID запущенной программы
ZOMBIE_PID=$!
echo "PID программы zombie: $ZOMBIE_PID"

# Ждем 3 секунды для создания зомби
sleep 3

# Ищем дочерний процесс
CHILD_PID=$(ps --ppid $ZOMBIE_PID -o pid= | head -1 | tr -d ' ')
echo "Дочерний процесс: $CHILD_PID"

# Проверяем статус дочернего процесса
echo "=== Проверка статуса процесса ==="
ps -o pid,state,command -p $CHILD_PID

# Если статус Z - это зомби
if ps -o state -p $CHILD_PID 2>/dev/null | grep -q "Z"; then
    echo "ОБНАРУЖЕН ЗОМБИ-ПРОЦЕСС (статус Z)"
else
    echo "Процесс не в статусе зомби"
fi

# Ждем завершения программы
wait $ZOMBIE_PID
echo "Программа zombie завершена"
```

**Что наблюдаем:**
- Создается дочерний процесс
- Дочерний процесс завершается, но становится зомби (статус Z)
- Родитель вызывает wait() и убирает зомби

# Теория

## ЗАДАНИЕ 1: Процессы и сигналы

### Теория

#### Что такое процесс?
**Процесс** - это программа в состоянии выполнения. У каждого процесса:
- Собственная память
- Отдельное адресное пространство  
- Уникальный PID (Process ID)
- Родительский процесс (PPID)

#### Сигналы в Linux
**Сигналы** - это уведомления, которые ОС отправляет процессам. Как сообщения "прервать работу", "завершиться" и т.д.

**Основные сигналы:**
- `SIGALRM` (14) - "будильник", срабатывает по таймеру
- `SIGKILL` (9) - "немедленно завершиться", нельзя игнорировать
- `SIGTERM` (15) - "вежливо завершиться"

#### Ключевые функции:

**`fork()`** - создает копию процесса:
```c
pid_t pid = fork();
if (pid == 0) {
    // Это дочерний процесс
} else {
    // Это родительский процесс
}
```

**`kill(pid, signal)`** - отправляет сигнал процессу:
```c
kill(child_pid, SIGKILL); // Принудительно завершить процесс
```

**`alarm(seconds)`** - устанавливает таймер:
```c
alarm(5); // Через 5 секунд придет SIGALRM
```

**`signal(signal, handler)`** - устанавливает обработчик сигнала:
```c
signal(SIGALRM, handle_alarm); // Вызовет handle_alarm при SIGALRM
```

**`waitpid(pid, &status, WNOHANG)`** - неблокирующее ожидание:
```c
// WNOHANG - не ждать, если процесс еще работает
waitpid(-1, &status, WNOHANG);
```

### Что происходит в вашей программе:

**Без timeout:**
1. Создаются 2 дочерних процесса
2. Каждый ищет min/max в массиве
3. Родитель ждет их завершения
4. Процессы завершаются нормально (статус 0)

**С timeout:**
1. Создаются 2 дочерних процесса  
2. Устанавливается таймер на 5 секунд
3. Если процессы не завершились за 5 сек - срабатывает SIGALRM
4. Обработчик отправляет SIGKILL дочерним процессам
5. Процессы принудительно завершаются (статус 9)

## ЗАДАНИЕ 2: Зомби-процессы

### Теория

#### Жизненный цикл процесса:
1. **Запуск** - процесс создан
2. **Выполнение** - процесс работает
3. **Завершение** - процесс закончил работу
4. **Зомби** - процесс завершен, но статус не прочитан

#### Что такое зомби-процесс?
**Зомби (Zombie)** - процесс, который завершил выполнение, но еще остается в таблице процессов, потому что родитель не прочитал его статус завершения.

**Почему появляются зомби?**
```c
// Родительский процесс
pid_t pid = fork();
if (pid == 0) {
    exit(0);  // Дочерний процесс завершается
} else {
    // Родитель НЕ вызывает wait() - появляется зомби!
    sleep(10);
}
```

#### Чем опасны зомби-процессы?
- Занимают записи в таблице процессов
- Система имеет ограниченное количество записей
- Много зомби могут "заблокировать" систему

#### Как убрать зомби?
1. **Родитель вызывает `wait()`** - читает статус завершения
2. **Родитель завершается** - все потомки "усыновляются" init процессом

### Что происходит в вашей программе:

1. **Создается дочерний процесс** (PID 29221)
2. **Дочерний процесс завершается** через 2 секунды
3. **Родитель спит 10 секунд** и НЕ вызывает wait()
4. **Дочерний процесс становится зомби** (статус Z)
5. **Родитель просыпается и вызывает wait()** - зомби исчезает

**В вашем выводе:**
```
PID S COMMAND
29221 Z [zombie] <defunct>
```
✅ **Z** - статус зомби-процесса

## ЗАДАНИЕ 3: Виртуальная память

### Теория

#### Что такое виртуальная память?
Каждому процессу кажется, что у него есть вся память компьютера. На самом деле ОС мапирует виртуальные адреса на физические.

#### Сегменты памяти процесса:

**1. Text (Код)**
- Исполняемые инструкции программы
- Только для чтения
- **`etext`** - адрес конца сегмента кода

**2. Data (Данные)**
- Инициализированные глобальные переменные
- Можно читать и писать
- **`edata`** - адрес конца сегмента данных

**3. BSS (Block Started by Symbol)**
- Неинициализированные глобальные переменные
- Инициализируются нулями при запуске
- **`end`** - адрес конца сегмента BSS

**4. Heap (Куча)**
- Динамическая память (malloc/free)
- Растет вверх

**5. Stack (Стек)**
- Локальные переменные, параметры функций
- Растет вниз

### Что происходит в вашей программе:

**В вашем выводе:**
```
Address etext: 3902A511     ← Конец кода
Address edata: 3902D018     ← Конец данных  
Address end  : 3902D050     ← Конец BSS

ID main    is at virtual address: 3902A249   ← Функция main в коде
ID cptr    is at virtual address: 3902D010   ← Глобальная переменная в данных
ID buffer1 is at virtual address: 3902D030   ← Глобальный массив в данных
ID i       is at virtual address: 10006404   ← Локальная переменная в стеке
Alocated memory at 685E06B0                  ← Динамическая память в heap
```

**Адреса показывают:**
- Код (text) занимает адреса до 3902A511
- Данные (data) от 3902D010 до 3902D018
- BSS от 3902D030 до 3902D050
- Стек вокруг 10006404 (обычно в верхней памяти)
- Heap вокруг 685E06B0 (растет вверх)

## ЗАДАНИЕ 5: Параллельное суммирование

### Теория

#### Процессы vs Потоки

**Процессы:**
- Изолированная память
- Создание через fork()
- Тяжеловесные
- Общение через IPC

**Потоки (Threads):**
- Разделяют память процесса
- Создание через pthread_create()
- Легковесные
- Общие глобальные переменные

#### POSIX Threads (pthreads)

**POSIX Threads (pthreads)** - это стандарт для работы с **потоками** (threads) в UNIX-подобных операционных системах.

Простыми словами: **"Это способ заставить программу делать несколько дел одновременно внутри одного процесса"**

**Создание потока:**
```c
pthread_t thread;
pthread_create(&thread, NULL, function, argument);
```

**Ожидание завершения:**
```c
void* result;
pthread_join(thread, &result);
```

#### Преимущества многопоточности:
- **Ускорение** - несколько ядер работают одновременно
- **Отзывчивость** - UI не блокируется при вычислениях
- **Эффективность** - меньше накладных расходов чем процессы

#### Проблемы многопоточности:
- **Гонки данных** - несколько потоков пишут в одну переменную
- **Взаимные блокировки** - потоки ждут друг друга
- **Сложность отладки**

### Что происходит в вашей программе:

**Разделение работы:**
```c
// Для 2 потоков и массива из 100 элементов:
Поток 1: элементы 0-49
Поток 2: элементы 50-99
```

**В ваших результатах:**
```
1 поток:  0.000197 сек
2 потока: 0.000150 сек  ← Быстрее на 31%!
4 потока: 0.000342 сек  ← Медленнее (oversubscription)
```

**Почему 4 потока медленнее?**
- У вас 2 ядра в Codespaces
- 4 потока создают больше переключений контекста
- Настройка и синхронизация потоков занимает время

**Корректность:**
```
Seed 100, 1000 элементов: сумма = 49511
```
✅ Одинаковый seed = одинаковый массив = одинаковая сумма

# Дополнительно

## Неблокирующий `waitpid()` с WNOHANG

Родитель периодически **проверяет** состояние дочерних процессов без блокировки. Использует флаг `WNOHANG` (Wait No Hang), который означает "не зависай".

```python
// Цикл с waitpid() + WNOHANG
while (waitpid(pid, &status, WNOHANG) == 0) {
    // Делайте полезную работу
    process_events();
    sleep(0.1);
}
```


## Обработка сигнала SIGCHLD

Операционная система отправляет родительскому процессу сигнал `SIGCHLD` когда любой из его дочерних процессов изменяет состояние (завершается, останавливается, продолжается).

```python
// SIGCHLD handler
void cleanup_child(int sig) {
    while (waitpid(-1, NULL, WNOHANG) > 0);
}
signal(SIGCHLD, cleanup_child);

```

## Двойной fork

Создается промежуточный процесс-"изолятор", который сразу завершается, оставляя рабочий процесс "сиротой". Сиротские процессы автоматически усыновляются процессом `init` (PID 1), который всегда вызывает `wait()` для своих детей.


```python
// Двойной fork()
if (fork() == 0) {
    if (fork() == 0) {
        // Рабочий процесс
        work();
    }
    exit(0);  // Изолятор завершается
}
wait(NULL);  // Ждем только изолятора
```

```
Оригинальный родитель (PID: 1000)
    ↓ fork()
Изолятор (PID: 1001) ← быстро завершается
    ↓ fork()  
Рабочий процесс (PID: 1002) ← СТАНОВИТСЯ СИРОТОЙ
    ↓ усыновляется
init (PID: 1) ← автоматически вызывает wait()
```
