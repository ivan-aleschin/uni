
Тестирование программного обеспечения – процесс исследования, испытания программного продукта, имеющий две различные цели: продемонстрировать разработчикам и заказчикам, что программа соответствует требованиям; выявить ситуации, в которых поведение программы является неправильным, нежелательным или не соответствующим спецификации

---
# Ключевые определения 

- Тестирование ПО — процесс исследования и испытаний ПО с целями: 1) доказать соответствие требованиям; 2) выявить некорректное/нежелательное поведение.
- Функциональное тестирование — проверка соответствия приложения функциональным требованиям (обычно black‑box).
- Black‑box (чёрный ящик) — тестирование без знания внутренней структуры, тесты строятся на основании требований/спецификаций.
- White‑box (белый ящик) — тестирование с учётом внутренней структуры (покрытие операторов, условий, ветвлений).
- Gray‑box — комбинированный подход (частичный доступ к реализации).

# Стратегии тестирования

- Black‑box: основан на спецификации, применяется на уровне системного/приёмочного тестирования; не требует знания кода; выполняет тестировщик.
- White‑box: основан на проекте/коде; применяется на уровне модулей/интеграции; требует доступа к коду; часто выполняют разработчики.
- Gray‑box: комбинация — полезно при интеграционном тестировании.

# Стандарт

Стандарт ISO/IEC 25010:2011 (ГОСТ Р ИСО/МЭК 25010-2015) определяет качество программного обеспечения как степень удовлетворения системой заявленных и подразумеваемых потребностей различных заинтересованных сторон. В соответствии со стандартом модель качества продукта включает восемь характеристик: 
- функциональная пригодность
- уровень производительности
- совместимость 
- удобство пользования
- надёжность
- защищённость
- сопровождаемость
- переносимость (мобильность)

# Функциональное тестирование (black‑box)

- Цель: проверить, что поведение системы соответствует требованиям.
- На что обращать внимание: функции, интерфейсы, взаимодействие с БД, инициализация/завершение, обработка ошибок/валидаторов.
- Ограничение: exhaustive testing невозможен — применяются техники выбора тестов.

# Тестовые артефакты (обязательно для отчёта/защиты)

- Test Plan (план тестирования): объект тестирования, стратегия, критерии входа/выхода, расписание, окружение, риски.
- Test Suite / Test Case (набор тест‑кейсов и отдельные кейсы).
- Bug Report (репорт дефекта): шаги воспроизведения, ожидаемый/фактический результат, окружение, приоритет.
- Test Report (отчёт): что проверено, результаты, покрытие, открытые баги.
- Чек‑листы, сценарии на основе use‑case (преобразование прецедентов в тесты).

# Техники тест‑дизайна (с практическими правилами и примерами)

1. Эквивалентные классы (Equivalence Partitioning)

- Идея: разбить множество входных данных на классы, внутри класса поведение одинаково → достаточно 1 теста из класса.
- Правила (основные):
    - Для диапазона n..m: один допустимый класс [n..m], два недопустимых: <n и >m.
    - Для фиксированного значения a: допустимый {a}, недопустимые <a и >a.
    - Для множества {a,b,c}: допустимый {a,b,c}, недопустимый — все остальные.
    - Для булевых — два класса: true/false.
    - Учесть группы переменных, влияющих совместно.
- Пример: поле «возраст» 18..65 → классы: допустимый [18..65], недопустимый <18, недопустимый >65.

2. Анализ граничных значений (Boundary Value Analysis)

- Идея: большинство ошибок на границах диапазонов. Тестируем минимальное/максимальное и «соседние» значения (min, min-1, min+1, max-1, max, max+1).
- Правило: для каждого диапазона тестировать обе границы и значения рядом с ними.
- Пример: возраст 18..65 → тесты: 17, 18, 19, 64, 65, 66.

3. Причинно‑следственный анализ (Cause‑Effect / Decision Table / Pairwise)

- Идея: моделируем связи «причины» (входные условия) → «следствия» (выход/действие). Полезно для сложной логики с комбинациями условий.
- Формат: таблица решений или диаграмма причин‑следствий. Пример: добавление клиента: поля Имя, Телефон, Email — комбинации валидности приводят к разным действиям (успех/ошибка/запрос подтверждения).
- Pairwise/комбинаторное тестирование: покрываем все пары значений (полезно, когда много параметров).

4. Предугадывание ошибок (Error Guessing)

- Идея: использовать опыт и здравый смысл, думать, где обычно делают ошибки (null, пустые строки, пробелы, очень длинные значения, спецсимволы, неправильные форматы).
- Примеры: не вводить обязательное поле; вводить SQL‑инъекцию; использовать максимальную длину поля; вводить неправильную локаль/кодировку.

5. Исчерпывающее тестирование

- Проверка всех возможных комбинаций; практически нереализуемо при ненулевой сложности; применимо в очень узких/малых доменах.

# Ответы на контрольные вопросы

1. В чем заключается задача тестирования ПО?

	- Задача тестирования — подтвердить соответствие программы требованиям и выявить случаи, в которых поведение программы является неправильным, нежелательным или не соответствующим спецификации. Тестирование помогает снизить риски, улучшить качество и обеспечить соответствие заявленным/подразумевающимся потребностям заинтересованных сторон.

2. Что такое тестовые артефакты?

- Тестовые артефакты — документы/модели, создаваемые и использующиеся в процессе тестирования: Test Plan (план тестирования), Test Cases и Test Suites (набор тест‑кейсов), Bug Reports (репорты дефектов), Test Report (итоговый отчёт), чек‑листы, сценарии (use‑case→test case), таблицы тестовых данных и пр. Каждый артефакт содержит информацию, необходимую для планирования, выполнения, отслеживания и анализа тестирования.

3. Что такое техники тест‑дизайна?

- Техники тест‑дизайна — методы выбора тестовых данных и формулировки тестов из множества всех возможных проверок с целью эффективного обнаружения дефектов. Основные техники: эквивалентное разделение, анализ граничных значений, причинно‑следственный анализ (decision tables), предугадывание ошибок, исчерпывающее тестирование (в редких случаях).

4. Что такое тестовый сценарий?

- Тестовый сценарий (test case) — формализованное описание одной проверки: цель, предпосылки, шаги выполнения, входные данные и ожидаемый результат. Хороший тест‑сценарий должен быть воспроизводимым, иметь чёткие критерии прохождения/провала, состоять из необходимых и достаточных действий, высокую вероятность обнаружения дефекта и не быть избыточным.
