# THEORY — Лабораторная работа №4: CYBERcobra (простейшее программируемое устройство)

## 0) Зачем эта лабораторная
ЛР4 показывает, как из простых блоков (АЛУ, регистровый файл, память инструкций, сумматор, счётчик команд) собирается рабочее программируемое устройство. Это «мост» между логикой (ЛР1–ЛР3) и полноценной архитектурой процессора.

После этой лабы ты должен:
- понимать, что такое процессор на уровне блок-схемы;
- уметь читать и писать простые машинные инструкции;
- понимать, как работает PC и ветвления;
- уметь проверить работу устройства по waveform.

---

## 1) Что такое программируемое устройство
Программируемое устройство — это цифровая схема, которая:
1) читает инструкции из памяти;
2) декодирует их;
3) управляет операциями над данными;
4) изменяет своё состояние (регистры, PC).

Ключевое отличие от «просто схемы»: здесь поведение задаётся программой.

---

## 2) Архитектура CYBERcobra: основные особенности
CYBERcobra — учебная архитектура с упрощениями:

- нет памяти данных (только регистровый файл);
- минимальная логика управления;
- инструкции фиксированной длины (32 бита);
- 5 типов инструкций:
  1) вычислительные (ALU);
  2) загрузка константы;
  3) загрузка с внешнего ввода;
  4) условный переход (branch);
  5) безусловный переход (jump).

Эта архитектура неэффективна по кодированию, но очень удобна для обучения.

---

## 3) Основной цикл выполнения инструкции
Типичный цикл работы:

1) PC указывает на адрес инструкции;
2) инструкция читается из памяти (IMEM);
3) поля инструкции декодируются;
4) АЛУ и регистры выполняют операцию;
5) PC обновляется (PC+4 или PC+offset*4);
6) следующий такт.

---

## 4) Роль программного счётчика (PC)
PC (Program Counter) — регистр, хранящий адрес текущей инструкции.

- Каждая инструкция занимает 4 байта;
- поэтому обычный шаг: PC = PC + 4.

Если это переход:
- branch: PC = PC + offset*4 (если условие истинно);
- jump: PC = PC + offset*4 всегда.

Важно: PC всегда кратен 4.

---

## 5) Формат инструкции (32 бита)
Инструкция делится на поля (некоторые пересекаются):

- J (бит 31) — безусловный переход;
- B (бит 30) — условный переход;
- WS (биты 29:28) — источник записи в регфайл;
- alu_op (27:23) — код операции АЛУ;
- RA1, RA2 (22:18 и 17:13) — адреса операндов;
- WA (4:0) — адрес записи;
- const (27:5) — 23-битная константа;
- offset (12:5) — 8-битное смещение (для переходов).

---

## 6) Источники записи в регистровый файл (WS)
WS управляет мультиплексором:

- WS = 00 → константа (const, знакорасширенная);
- WS = 01 → результат АЛУ;
- WS = 10 → внешние данные (sw_i);
- WS = 11 → не используется (обычно 0).

---

## 7) Логика WE (write enable)
В регистровый файл нельзя писать, если выполняется переход:

- если B = 1 или J = 1 → WE = 0;
- иначе WE = 1.

Это предотвращает случайную запись при ветвлениях.

---

## 8) Знакорасширение (Sign Extension)
Чтобы короткие константы стали 32-битными:

- const (23 бита):
  const[27:5] → {9 повторов старшего бита, затем const}

- offset (8 бит):
  offset[12:5] → {22 повтора старшего бита, offset, 2 нуля справа}

Два нуля справа — это умножение offset на 4 (адресация по словам).

---

## 9) Условный переход (Branch)
Для branch используется АЛУ как компаратор:

- ALU сравнивает reg[RA1] и reg[RA2] по alu_op;
- результат сравнения → flag;
- если flag = 1 → PC += offset*4;
- иначе → PC += 4.

---

## 10) Безусловный переход (Jump)
Если J = 1, переход всегда выполняется:

PC = PC + offset*4.

В этом случае WE = 0 (записи в регистры нет).

---

## 11) Вход и выход
- sw_i — внешний 16-битный ввод (переключатели на плате);
- out_o — 32-битный вывод (в CYBERcobra подключён к одному из регистров, обычно RA1).

Чтобы «зафиксировать» результат на выходе:
делается бесконечная инструкция перехода с offset = 0 и RA1 = адрес результата.

---

## 12) Программирование (как писать программы)
### Базовые принципы
- Всё хранится в регистрах.
- Нет памяти данных → только регфайл.
- Для циклов используют do…while (с переходом назад).
- Для if/else нужно помнить, что после branch всегда идёт else-блок.

### Пример if/else
Логика:
- после branch сразу блок else;
- в конце else ставится jump, чтобы перепрыгнуть if.

### Пример if без else
Часто проще инвертировать условие и перепрыгнуть блок if, если условие ложно.

---

## 13) Пример задачи: циклический сдвиг вправо
Задача: out = (a >> k) | (a << (32 - k)), где k = sw_i & 31.

Пошагово:
1) k = sw_i & 31
2) t1 = a >> k
3) t2 = a << (32 - k)
4) out = t1 OR t2

Это пример того, как из элементарных операций собирается более сложное действие.

---

## 14) Форматы файлов программы
### program.txt
Читабельный формат:
- допускаются комментарии и пробелы;
- в каждой строке после очистки должно быть 32 бита.

### program.mem
Формат HEX для $readmemh:
- каждая строка — 1 инструкция (32 бита);
- пример строки: 00000161.

---

## 15) cyberconverter (вспомогательная утилита)
Утилита превращает program.txt → program.mem:
- удаляет комментарии и пробелы;
- проверяет длину строки;
- переводит из двоичного в hex.

---

## 16) Проверка в симуляции
В GTKWave удобно смотреть:

- pc_q (адрес инструкции)
- instr (содержимое)
- ws, bit_b, bit_j (управление)
- rf_we (разрешение записи)
- ra1, ra2, wa (адреса)
- rf_wd (что пишется)
- alu_result, alu_flag
- pc_addend, pc_sum

Смысл: по тактам подтверждать, что PC идёт правильно, регистры пишутся правильно, ветвления работают.

---

## 17) Vivado и ПЛИС (кратко, но суть)
- Vivado превращает SystemVerilog в конфигурацию ПЛИС.
- Прошивка = bitstream.
- Схема работает параллельно, не «выполняет код».

Минимальный цикл:
1) Создать RTL проект.
2) Добавить *.sv как Design Sources.
3) Добавить testbench как Simulation Sources.
4) Добавить program.mem.
5) Behavioral Simulation (проверка).
6) Synthesis → Implementation → Generate Bitstream.
7) Program Device (прошивка).

---

## 18) Типовые ошибки
- Нет program.mem в проекте → IMEM пустая.
- Перепутаны поля инструкции (RA1/RA2/WA).
- Забыто знакорасширение.
- offset не умножен на 4.
- WE не отключён при ветвлениях.
