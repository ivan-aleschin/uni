# THEORY — Лабораторная работа №4: CYBERcobra (простейшее программируемое устройство)

## 0) Что это за лабораторная и зачем она
ЛР4 — это сборка «учебного процессора» из блоков, сделанных в ЛР1–ЛР3:
- сумматор (Adder),
- АЛУ (ALU),
- регистровый файл (Register File),
- память инструкций (Instruction Memory),
- счётчик команд (Program Counter, PC).

Цель — понять, как выполняются инструкции, как декодируются их поля и как логика управления связана с результатом работы АЛУ. После этой лабы структура RISC‑V будет интуитивно понятнее.

---

## 1) Архитектура CYBERcobra: идея
CYBERcobra — минималистичный процессор:
- **нет памяти данных** (все данные в регистровом файле),
- простая логика управления (фактически проводники + пара вентилей),
- 5 типов инструкций:
  1) вычислительные (ALU)
  2) загрузка константы
  3) загрузка с внешнего ввода
  4) условный переход (branch)
  5) безусловный переход (jump)

---

## 2) Роль программного счётчика (PC)
PC хранит адрес **текущей** инструкции в памяти.  
Каждая инструкция = 4 байта, значит следующий адрес обычно:

    PC = PC + 4

Если выполняется переход:
- **branch**: PC = PC + offset*4 (если условие истинно)
- **jump**: PC = PC + offset*4 (всегда)

---

## 3) Поля инструкции (32 бита)
В CYBERcobra все инструкции — 32‑битные. Поля накладываются, но логика выбирает нужные.

Ключевые поля:
- `J` (бит 31): безусловный переход
- `B` (бит 30): условный переход
- `WS` (биты 29:28): источник записи в регистровый файл
- `alu_op` (биты 27:23): операция АЛУ
- `RA1`, `RA2` (биты 22:18, 17:13): адреса операндов
- `WA` (биты 4:0): адрес записи
- `const` (биты 27:5): 23‑битная константа
- `offset` (биты 12:5): 8‑битное смещение (для переходов)

---

## 4) Логика записи в регистровый файл
Запись есть только в инструкциях **не-перехода**.

### Логика `WE`:
- если `B = 1` или `J = 1` → **WE = 0**
- иначе → **WE = 1**

### Источник записи (WS):
- `00` → константа из инструкции (`const`, знакорасширенная)
- `01` → результат АЛУ
- `10` → `sw_i` (внешний ввод)
- `11` → не используется (можно писать 0)

---

## 5) Переходы (Branch/Jump)

### Условный переход (B = 1)
АЛУ выполняет операцию сравнения (`alu_op`), результат → `flag`:
- если `flag = 1`: PC += offset*4
- если `flag = 0`: PC += 4

### Безусловный переход (J = 1)
PC += offset*4 (без проверок).

### Смещение (offset)
- `offset` хранится в **битах [12:5]**
- это число **в инструкциях**, а не в байтах
- поэтому на аппаратном уровне:  
  `offset*4` = `{offset, 2’b00}`  
- необходимо **знакорасширить** offset до 32 бит.

---

## 6) Знакорасширение (Sign Extension)
### Для `const` (23 бита):
    const[27:5] → {{9{const[22]}}, const}

### Для `offset` (8 бит):
    offset[12:5] → {{22{offset[7]}}, offset, 2’b00}

---

## 7) Вход/выход
- `sw_i` — 16‑битный внешний ввод (обычно переключатели на плате)
- `out_o` — вывод процессора (подключён к `reg_file[RA1]`)

Чтобы вывести результат, в конце программы делают инструкцию:

    J=1, offset=0, RA1 = адрес результата

Это даёт бесконечный цикл с отображением нужного регистра.

---

## 8) Как писать программы
### Общая идея
- Все операции — через регистры.
- Нет памяти данных, значит всё хранится в регфайле.

### Условные блоки
Для `if/else`:
- после branch сразу идёт **else** блок
- в конце else нужен `jump` через if‑блок

Для `if` без else:
- можно инвертировать условие, чтобы прыгнуть через блок.

### Циклы
Удобно делать `do...while`:
- тело цикла,
- условный переход назад.

---

## 9) Пример: циклический сдвиг вправо
Псевдокод:

    k = sw_i & 31
    res = (a >> k) | (a << (32 - k))

Для этого нужны:
- AND (вычисление k)
- SUB (32 - k)
- SRL / SLL
- OR

---

## 10) Форматы файлов
### program.txt
Формат читаемый (с комментариями):

    //J B WS ALUop RA1 RA2 offset WA
      0 0 00 ... ... ... ... ...

Комментарии и пробелы допустимы.

### program.mem
Формат HEX для `$readmemh`:

    00000161
    20000002
    ...

Одна строка = 32 бита = 1 инструкция.

---

## 11) cyberconverter
`cyberconverter` превращает `program.txt` в `program.mem`:
- удаляет комментарии/пробелы
- проверяет длину 32 бита
- конвертирует в HEX

---

## 12) Проверка в симуляции (GTKWave)
Полезные сигналы:
- `pc_q` — адрес текущей инструкции
- `instr` — прочитанная инструкция
- `ws`, `bit_b`, `bit_j` — управляющие биты
- `rf_we` — запись в регфайл
- `ra1`, `ra2`, `wa` — адреса регистров
- `rf_wd` — что пишется
- `alu_result`, `alu_flag`
- `pc_addend`, `pc_sum`

Смысл проверки: по тактам видеть, какие инструкции выполняются и куда идёт PC.

---

## 13) Vivado и ПЛИС (FPGA)
**Vivado** — это среда синтеза и прошивки ПЛИС.  
Минимальный порядок:

1. Создать RTL проект.
2. Добавить все `*.sv` как Design Sources.
3. Добавить testbench как Simulation Source.
4. Добавить `program.mem`.
5. Запустить Behavioral Simulation.
6. Синтез → Implementation → Generate Bitstream.
7. Program Device.

---

## 14) Типичные ошибки
- Не добавлен `program.mem` в проект → память пуста.
- Перепутаны поля инструкции (RA1/RA2/WA).
- Забыто знакорасширение для `const` или `offset`.
- Не учтено `offset*4` (две нули справа).
- `WE` не выключается на ветвлениях.
