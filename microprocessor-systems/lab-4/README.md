## Что должно работать

- **Вычислительные инструкции** → запись результата АЛУ в регистр
- **Загрузка константы** → запись `rf_const` в регистр
- **Загрузка с внешних устройств** → запись `sw_i` в регистр
- **Условные переходы** → PC меняется только если `flag == 1`
- **Безусловные переходы** → PC всегда смещается
- При инструкциях перехода `WE = 0`

## Индивидуальное задание — подсказка по инициализации и формату `program.mem`

Эта часть относится именно к индивидуальному заданию (программа для CYBERcobra).

### Почему `program.mem` выглядит «странно»
`program.mem` — это **HEX‑формат** для `$readmemh`, по 1 инструкции (32 бита) на строку.
Поэтому строки выглядят как `00000161`, `20000002` и т.д. — это просто шестнадцатеричное представление 32‑битных инструкций.

Процесс:
1) Пишешь программу в `program.txt` (читаемый формат с 0/1 и комментариями)
2) Запускаешь `cyberconverter program.txt program.mem` → получаешь `program.mem` в HEX
3) `instr_mem` читает HEX через `$readmemh`

Примечание: `cyberconverter` — отдельная утилита из материалов APS, это не встроенная функция Vivado. На NixOS его обычно собирают из `cyberconverter.cpp` (например, через `g++`), либо используют готовый бинарник под свою ОС.

### Подробная расшифровка `program.txt` (вариант 1)
Формат полей в каждой строке: `J B WS alu_op/const RA1 RA2 offset WA`.

- `J` — безусловный переход
- `B` — условный переход
- `WS` — источник записи в регфайл
- `alu_op` — код операции АЛУ (для ALU-инструкций)
- `const` — 23‑битная константа (для загрузки константы)
- `RA1`, `RA2` — адреса операндов
- `offset` — смещение (для переходов)
- `WA` — адрес записи результата

Адреса регистров указаны в двоичном виде:  
`00001` = r1, `00010` = r2, `00011` = r3, ... `01001` = r9.

| № | J | B | WS | alu_op / const | RA1 | RA2 | offset | WA | Действие |
|---|---|---|----|----------------|-----|-----|--------|----|---------|
| 1 | 0 | 0 | 00 | const = 000...01011 | — | — | — | 00001 | r1 = 11 |
| 2 | 0 | 0 | 10 | const = 000...00000 | — | — | — | 00010 | r2 = sw_i |
| 3 | 0 | 0 | 00 | const = 000...11111 | — | — | — | 00011 | r3 = 31 |
| 4 | 0 | 0 | 01 | alu_op = 00111 (AND) | 00010 | 00011 | 00000000 | 00100 | r4 = r2 & r3 |
| 5 | 0 | 0 | 00 | const = 000...100000 | — | — | — | 00101 | r5 = 32 |
| 6 | 0 | 0 | 01 | alu_op = 01000 (SUB) | 00101 | 00100 | 00000000 | 00110 | r6 = r5 - r4 |
| 7 | 0 | 0 | 01 | alu_op = 00101 (SRL) | 00001 | 00100 | 00000000 | 00111 | r7 = r1 >> r4 |
| 8 | 0 | 0 | 01 | alu_op = 00001 (SLL) | 00001 | 00110 | 00000000 | 01000 | r8 = r1 << r6 |
| 9 | 0 | 0 | 01 | alu_op = 00110 (OR) | 00111 | 01000 | 00000000 | 01001 | r9 = r7 \| r8 |
|10 | 1 | 0 | 00 | alu_op = 00000 | 01001 | 00000 | 00000000 | 00000 | бесконечный jump, out_o = r9 |

Почему последняя строка выводит результат:  
при `J=1` и `offset=0` PC остаётся на этой инструкции.  
`RA1 = r9`, а `out_o` подключён к `read_data1` регфайла → на выходе постоянно r9.

### Пример настройки для варианта 1
Для варианта 1 (циклический сдвиг вправо) выбрана простая константа `a = 11` (`0x0000000B`), чтобы было легко проверить результат вручную.
Вход `sw_i` в `testbench.sv` установлен как в примере — `2`.

Ожидаемый результат:
- `k = sw_i & 31 = 2`
- `out_o = (a >> k) | (a << (32 - k)) = 0xC0000002`

Исходник программы лежит в `program.txt`.  
Файл `program.mem` сгенерирован из него (формат HEX для `$readmemh`).
