# Лабораторная работа №6 — Процессорная система RISC-V с LSU и прерываниями

Данная работа является финальным этапом интеграции процессорного ядра. Она объединяет три ключевых модуля из репозитория APS: **ЛР7 (Тракт данных)**, **ЛР9 (LSU)** и **ЛР11 (Прерывания)**.

---

## 🏗️ Архитектура и Схема

Система построена по классической однотактной схеме RISC-V с расширениями для работы с памятью и обработки внешних событий.

### 1. Тракт данных (Datapath - ЛР7)
Это "скелет" процессора. Включает в себя:
- **Core (Ядро):** Координирует работу всех блоков.
- **Decoder:** Превращает 32-битную инструкцию в набор управляющих сигналов (`a_sel`, `b_sel`, `alu_op`, `we` и др.).
- **ALU (АЛУ):** Выполняет арифметику (ADD, SUB), логику (AND, OR) и сравнения.
- **Register File:** 32 регистра по 32 бита. Регистр `x0` аппаратно зафиксирован в 0.
- **PC Logic:** Рассчитывает адрес следующей инструкции:
    - `PC + 4` для обычных команд.
    - `PC + imm` для переходов (`branch`, `jal`).
    - `rs1 + imm` для `jalr`.
    - `mtvec` при возникновении прерывания.
    - `mepc` при возврате из прерывания (`mret`).

### 2. LSU (Load-Store Unit - ЛР9)
Модуль-посредник между ядром и памятью данных.
- **Зачем нужен:** Процессор работает со словами (32 бита), но инструкции `lb` (byte) или `lh` (halfword) требуют работы с частями слова. LSU выделяет нужные байты из памяти при чтении и маскирует их при записи.
- **Stall Logic:** Поскольку блочная память ПЛИС выдает данные с задержкой в 1 такт, LSU выставляет сигнал `stall_o`, который "замораживает" `PC` и `IF/ID` регистры ядра на один цикл.

### 3. Подсистема прерываний (Interrupts - ЛР11)
Добавляет процессору "реактивность".
- **CSR Controller:** Содержит системные регистры:
    - `mtvec`: адрес начала обработчика прерываний.
    - `mepc`: адрес инструкции, на которой прервались (для возврата).
    - `mcause`: код причины прерывания.
    - `mie`: маска разрешенных прерываний.
    - `mstatus`: глобальный статус (бит MIE разрешает/запрещает прерывания).
- **Interrupt Controller:** Слушает внешний сигнал `irq_req`. Если прерывания разрешены, он посылает сигнал `trap` в ядро.

## 📈 Как проверить работу (Временная диаграмма)

При защите лабы в **GTKWave** или **Vivado Waveform** нужно смотреть на следующие этапы:

### 1. Обычное выполнение (Fetch -> Execute)
- Проверьте, что `PC` (сигнал `instr_addr_o`) меняется каждые 4 байта.
- Убедитесь, что `instr_i` соответствует коду из `program.mem`.
- Смотрите на `alu_result`: он должен меняться сразу после появления инструкции (комбинационная логика).

### 2. Обращение к памяти (Load/Store)
- При выполнении `lw` или `sw` должен появиться сигнал `mem_req_o = 1`.
- **Критический момент:** В этот же такт должен подняться `stall_i = 1`.
- `PC` должен остаться прежним на один такт.
- На следующем такте `stall_i` падает в 0, и данные считанные из `mem_rd_i` попадают в регистровый файл.

### 3. Обработка прерывания (Trap)
- Найдите момент, когда `irq_req_i` переходит в 1.
- Проверьте, что на следующем такте `PC` прыгает не на `PC+4`, а на адрес, записанный в `mtvec` (обычно это начало обработчика в `program.mem`).
- Сигнал `trap` в ядре должен быть равен 1 в момент прыжка.
- В регистр `mepc` должен записаться адрес инструкции, которая была прервана.

### 4. Возврат из прерывания (MRET)
- В конце обработчика выполняется инструкция `mret` (код `30200073`).
- Убедитесь, что после неё `PC` возвращается к значению из `mepc`.
- Сигнал `irq_ret_o` должен кратковременно подняться в 1.
