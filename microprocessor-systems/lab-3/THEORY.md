# THEORY — Лабораторная работа №3
**Тема:** регистровый файл и память инструкций  
**Цель:** подробно разобраться, чем отличается память инструкций (IMEM) от регистрового файла (RF), как они работают по такту, и как это описывается на SystemVerilog.

---

## 1) Что мы строим и зачем это нужно
В ЛР3 создаются два разных типа памяти:

- **Память инструкций (Instruction Memory, IMEM)**  
  Хранит программу. Процессор читает инструкции по адресу PC.

- **Регистровый файл (Register File, RF)**  
  Хранит операнды и результаты вычислений.

Оба блока — фундамент любой процессорной схемы: без IMEM нет инструкций, без RF нет быстрых данных для АЛУ.

---

## 2) IMEM vs RF — ключевые различия (таблица)

| Характеристика | IMEM (Память инструкций) | RF (Регистровый файл) |
|---|---|---|
| Назначение | Хранение программы | Хранение данных/операндов |
| Чтение | Асинхронное | Асинхронное |
| Запись | Нет (ROM) | Есть, синхронная |
| Порты | 1 порт чтения | 2 порта чтения + 1 запись |
| Инициализация | `$readmemh` из `program.mem` | Значения формируются в процессе работы |
| Размер | Обычно 128 слов (512 байт) | 32 регистра × 32 бита |
| Регистр нуля | Не актуально | `x0` всегда 0 |

---

## 3) Почему IMEM и RF устроены по‑разному
### IMEM — асинхронная ROM
Инструкция нужна **в том же такте**, когда известен PC.  
Поэтому чтение IMEM асинхронное: `read_data` меняется сразу при изменении адреса.

### RF — чтение асинхронно, запись синхронна
АЛУ получает два операнда сразу — значит нужны 2 независимых порта чтения.  
Запись по фронту такта защищает от гонок и делает поведение предсказуемым.

---

## 4) Аппаратный ноль (x0)
В архитектуре RISC‑V регистр `x0` всегда равен 0.  
Зачем это нужно:
- `x1 = x2 + x0` — копирование
- сравнение с нулём без отдельного регистра

Реализация в ЛР3:
- запись в `x0` запрещена
- чтение `x0` всегда даёт 0

---

## 5) Адресация памяти: почему `addr[10:2]`
IMEM хранит 32‑битные слова (4 байта).  
Адрес задаётся в **байтах** (byte addressing).

Поэтому индекс слова = `addr / 4`, а в бинарной форме это:

`addr[10:2]` — два младших бита отбрасываются.

Пример:
- адрес 0 → слово 0
- адрес 4 → слово 1
- адрес 8 → слово 2

---

## 6) `$readmemh` — как загружается программа
`$readmemh("program.mem", ROM);`

Файл `program.mem`:
- каждая строка = 32 бита (1 слово)
- записано в HEX
- если строк меньше массива — симулятор выдаёт предупреждение

Пример строки: `00000013`

---

## 7) SystemVerilog‑синтаксис (подробно)

### 7.1 Объявление памяти
`logic [31:0] ROM [0:WORDS-1];`

Это массив:
- каждый элемент = 32 бита
- всего `WORDS` слов

### 7.2 Асинхронное чтение (IMEM)
`assign read_data_o = ROM[addr_index];`

`assign` означает:
- **комбинаторная логика**
- выход меняется сразу при смене адреса

### 7.3 Синхронная запись (RF)
`always_ff @(posedge clk_i) begin ... end`

Запись происходит **только по фронту** `clk_i`.

### 7.4 Почему `logic`, а не `wire` и `reg`
`logic` — универсальный тип:
- может быть входом/выходом
- подходит и для комбинационной, и для последовательной логики

Это стандартный стиль в SystemVerilog.

---

## 8) Порты регистрового файла
RF имеет:
- `read_addr1_i` → `read_data1_o`
- `read_addr2_i` → `read_data2_o`
- `write_addr_i`, `write_data_i`, `write_enable_i`

Это означает:
- читаем два регистра сразу
- записываем один регистр по фронту

---

## 9) Как проверять в GTKWave
### Что смотреть:
- `clk`
- `write_enable`
- `write_addr`, `write_data`
- `read_addr1`, `read_addr2`
- `read_data1`, `read_data2`

### Ожидаемое поведение:
- при смене `read_addr` данные меняются сразу
- запись появляется **только после фронта clk**
- `read_data` для адреса 0 всегда 0
