# THEORY — Лабораторная работа №3
**Тема:** регистровый файл и память инструкций  
**Цель:** понять, как в процессоре устроена память команд и регистровый файл, а также закрепить навыки описания памяти на SystemVerilog.

---

## 1) Что это за блоки и зачем они нужны

### Память инструкций (Instruction Memory, IMEM)
- Хранит программу.
- Только чтение (Read Only Memory, ROM).
- Чтение **асинхронное**: данные появляются сразу при изменении адреса.
- Содержимое подгружается при старте моделирования через `$readmemh`.

**Почему так:** в однотактном процессоре инструкция нужна в тот же такт, когда адрес известен, поэтому чтение без ожидания фронта.

### Регистровый файл (Register File, RF)
- Быстрая память внутри процессора: **32 регистра по 32 бита**.
- Два независимых порта чтения (асинхронные).
- Один порт записи (синхронный, по фронту `clk`).
- Регистр `x0` (или `r0`) всегда равен нулю — это аппаратный ноль.

**Почему так:** в RISC‑V (и не только) регистр нуля позволяет писать компактные программы (например, сравнения с нулём или копирование через `+ 0`).

---

## 2) Адресация памяти

### Побайтовая адресация
В архитектуре используется **byte addressing**: каждый байт имеет свой адрес.  
Но слово — 32 бита = 4 байта. Поэтому:
```
index = addr[10:2]
```
То есть два младших бита адреса отбрасываются (адрес всегда кратен 4).

---

## 3) SystemVerilog: что используется в лабе

### 3.1 Массив памяти
```systemverilog
logic [31:0] ROM [0:WORDS-1];
```

### 3.2 Асинхронное чтение
```systemverilog
assign read_data_o = ROM[addr_index];
```

### 3.3 Синхронная запись
```systemverilog
always_ff @(posedge clk_i) begin
  if (we) mem[wa] <= wd;
end
```

### 3.4 Аппаратный ноль (x0)
Два варианта:
1) Запрет записи в `0` + подмена чтения на ноль  
2) Жёсткая инициализация `rf[0]=0` каждый такт  
В лабе используется вариант 1.

---

## 4) Формат `program.mem`

`$readmemh` читает **HEX**-строки по 32 бита:

```
01234567
89ABCDEF
...
```

- 1 строка = 1 слово (32 бита).
- Если строк меньше, Vivado/iverilog может выдавать предупреждение.
- Для чистой симуляции лучше заполнить оставшееся нулями.

---

## 5) Типовые ошибки

✅ **Частые проблемы:**
- Смешивание асинхронного чтения и синхронного — неправильный тайминг.
- Забыли запретить запись в `x0`.
- Не учли `addr[10:2]` → чтение идёт не по тем адресам.
- В Vivado не добавлен `program.mem` в проект.

✅ **Норма для тестов:**
- `read_data` меняется сразу, как только поменялся адрес.
- `write_data` попадает в память **только на фронте** `clk`.

## 6) Проверка в симуляции

### GTKWave сигналы, за которыми обычно следят
- `clk`
- `write_enable`
- `write_addr`
- `write_data`
- `read_addr1`, `read_addr2`
- `read_data1`, `read_data2`

Проверка:
- После фронта `clk` значение записывается.
- При смене `read_addr` данные меняются мгновенно.
