# Лабораторная работа №3 — Регистровый файл и память инструкций

Теория: `THEORY.md`

## Теория

Процессору нужна память двух видов:

**Память инструкций (ПЗУ)** — хранит программу. Только чтение. Асинхронный порт (данные появляются мгновенно по адресу, без ожидания такта). Содержимое загружается при старте через `$readmemh` из файла `program.mem`.

**Регистровый файл (ОЗУ)** — маленькая быстрая память внутри процессора. 32 регистра по 32 бита. Три порта: два асинхронных на чтение, один синхронный на запись. Данные сразу доступны по адресу чтения (без ожидания такта), запись происходит по фронту тактового сигнала.

### Регистр 0 — аппаратный ноль

По спецификации RISC-V регистр `x0` всегда равен нулю. Запись в него игнорируется, чтение всегда возвращает 0. Это позволяет упрощать многие операции: сравнение с нулём, инициализация, копирование через сложение с нулём.

Реализуется двумя способами (в этой лабе выбран первый):
- мультиплексор на выходе: если адрес == 0, вернуть 0
- запрет записи в ячейку 0 + инициализация нулём

### Байтовая адресация

RISC-V использует побайтовую адресацию: каждый байт в памяти имеет свой адрес. Но ячейки памяти 32-битные (4 байта). Поэтому адрес байта переводится в индекс ячейки отбрасыванием двух младших бит:

```
индекс_ячейки = addr[10:2]  // для 512-байтной памяти
```

## Пайплайн №1 — Разработка и проверка дома (iverilog + GTKWave)

### Шаг 1. Свой тестбенч

```bash
iverilog -g2012 -o sim memory_pkg.sv design.sv testbench.sv && vvp sim
gtkwave wave.vcd
```

Успех: `=== Тест завершён. Ошибок: 0 ===`

### Шаг 2. Официальный тест из репо

Официальный тест не использует SVA и `std::randomize` — он работает с iverilog. Но нужно добавить `` `timescale `` в начало файла перед `module`:

```systemverilog
`timescale 1ns/1ps  // добавить эту строку
module lab_03_tb_register_file();
```

Затем:

```bash
iverilog -g2012 -o sim memory_pkg.sv design.sv lab_03.tb_register_file.sv && vvp sim
```

Успех: `Number of errors: 0`

> `instr_mem` не тестируется официальным тестом — он проверяется только визуально в Vivado.

### Что проверяем в GTKWave

- `clk` — тактовый сигнал
- `we`, `wa`, `wd` — сигналы записи
- `ra1`, `ra2`, `rd1`, `rd2` — порты чтения
- Видно: запись происходит по фронту `clk`, чтение реагирует мгновенно на смену `ra1`/`ra2`
- `rd1`/`rd2` = 0 при адресе 0 (даже после попытки записи)

## Пайплайн №2 — Демонстрация в университете (Vivado)

### Шаг 1. Создать проект

1. **File → Project → New → RTL Project**
2. **Design Sources**: добавить все три файла:
   - `memory_pkg.sv`
   - `design.sv`
3. **Simulation Sources**: добавить `lab_03.tb_register_file.sv`
4. Добавить `program.mem` в проект (**Add Files** → выбрать файл)
5. Выбрать ту же ПЛИС, что и в ЛР1–2

### Шаг 2. Запустить симуляцию регистрового файла

1. В **Simulation Sources** правая кнопка на `lab_03_tb_register_file` → **Set as Top**
2. **Flow → Run Simulation → Run Behavioral Simulation**
3. В TCL-консоли: `Number of errors: 0`

### Шаг 3. Синтез и прошивка

1. В **Design Sources** правая кнопка на `register_file` → **Set as Top**
2. **Flow → Run Synthesis → Run Implementation → Generate Bitstream → Program Device**

### Что показывать преподавателю

- Временную диаграмму симуляции — запись синхронная, чтение асинхронное, `rd=0` при адресе 0
- В TCL-консоли: `Number of errors: 0`
- Работающую прошивку на плате
