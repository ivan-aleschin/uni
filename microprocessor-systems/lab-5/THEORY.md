# THEORY — Лабораторная работа №5
**Тема:** декодер инструкций RISC‑V (RV32I + Zicsr + `mret`)  
**Цель:** разобраться, как по 32‑битной инструкции формируются управляющие сигналы процессора, и как корректно обрабатывать нелегальные инструкции.

---

## 1) Роль декодера
Декодер — это комбинационная логика, которая по входной инструкции выдаёт набор управляющих сигналов для:
- АЛУ (какую операцию выполнять);
- мультиплексоров операндов АЛУ;
- интерфейса памяти данных;
- записи в регистровый файл;
- доступа к CSR.

Именно декодер решает: **что именно должна сделать инструкция**.

---

## 2) Форматы инструкций RISC‑V
Базовые форматы RV32I:

| Формат | Описание |
|---|---|
| R | операции над двумя регистрами |
| I | операции с непосредственным операндом (imm) |
| S | запись в память |
| B | ветвление |
| U | «длинные» immediates (LUI/AUIPC) |
| J | безусловный переход `jal` |

Декодирование начинается с `opcode` (биты `[6:2]`).  
Для большинства инструкций конкретная операция определяется полями `func3` и `func7`.

---

## 3) Поддерживаемые группы инструкций
Декодер поддерживает:
- вычисления (R‑тип, I‑тип);
- загрузку/сохранение (LOAD/STORE);
- ветвления и переходы (BRANCH/JAL/JALR);
- системные инструкции (CSR + `mret`).

Также обрабатывается `FENCE` как `NOP`.

---

## 4) Сигналы управления (что формируется декодером)

### 4.1 Сигналы источника операндов АЛУ
`a_sel_o`:
- `OP_A_RS1` — `rs1`
- `OP_A_CURR_PC` — PC
- `OP_A_ZERO` — ноль

`b_sel_o`:
- `OP_B_RS2` — `rs2`
- `OP_B_IMM_I` — I‑imm
- `OP_B_IMM_S` — S‑imm
- `OP_B_IMM_U` — U‑imm
- `OP_B_INCR` — константа 4 (PC+4)

### 4.2 Операции АЛУ
Коды заданы в `alu_opcodes_pkg.sv`.  
Они используются для:
- арифметики (ADD/SUB),
- логики (AND/OR/XOR),
- сдвигов (SLL/SRL/SRA),
- сравнений (EQ/NE/LTS/LTU/GES/GEU),
- операций SLT/SLTU.

### 4.3 Запись результата (wb_sel_o)
- `WB_EX_RESULT` — результат АЛУ
- `WB_LSU_DATA` — данные из памяти
- `WB_CSR_DATA` — данные из CSR

### 4.4 Интерфейс памяти
Для LOAD/STORE:
- `mem_req_o = 1` (обращение к памяти)
- `mem_we_o = 1` (только для STORE)
- `mem_size_o` (размер данных)

Коды `mem_size_o`:
| Значение | Описание |
|---|---|
| `LDST_B`  | знаковое 8‑битное |
| `LDST_H`  | знаковое 16‑битное |
| `LDST_W`  | 32‑битное |
| `LDST_BU` | беззнаковое 8‑битное |
| `LDST_HU` | беззнаковое 16‑битное |

### 4.5 Управление PC
- `branch_o` — условный переход
- `jal_o` — безусловный `jal`
- `jalr_o` — безусловный `jalr`
- `mret_o` — возврат из прерывания

---

## 5) Декодирование SYSTEM‑инструкций
Инструкции SYSTEM делятся на два класса:

### 5.1 CSR‑инструкции (Zicsr)
- `CSRRW`, `CSRRS`, `CSRRC`
- `CSRRWI`, `CSRRSI`, `CSRRCI`

Для них:
- `csr_op_o = func3`
- `csr_we_o = 1`
- `wb_sel_o = WB_CSR_DATA`
- `gpr_we_o = 1`

### 5.2 Специальные SYSTEM
- `ECALL`, `EBREAK` → считаются **нелегальными**  
- `MRET` → отдельная инструкция возврата

Полные 32‑битные коды:
- `ECALL`  = `32'h00000073`
- `EBREAK` = `32'h00100073`
- `MRET`   = `32'h30200073`

---

## 6) Нелегальные инструкции
Сигнал `illegal_instr_o = 1`, если:
1. младшие 2 бита `opcode` ≠ `11`;
2. opcode не входит в список поддерживаемых;
3. неправильные `func3`/`func7`;
4. `ECALL` / `EBREAK`;
5. некорректная SYSTEM‑инструкция.

При `illegal_instr_o = 1` **должны быть отключены все изменения архитектурного состояния**:
- `gpr_we_o = 0`
- `csr_we_o = 0`
- `mem_req_o = 0`
- `mem_we_o = 0`
- `branch_o = 0`
- `jal_o = 0`
- `jalr_o = 0`
- `mret_o = 0`

---

## 7) Идея реализации
Самый удобный стиль:
1. В `always_comb` задать **значения по умолчанию** (нелегальная инструкция).
2. По `opcode` определить группу инструкций.
3. Внутри по `func3`/`func7` — конкретную операцию.
4. В случае ошибки — оставить `illegal_instr_o = 1`.

Это минимизирует код и избегает защёлок.
