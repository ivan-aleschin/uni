## Предмет: Микропроцессорные средства и системы

Курс посвящён созданию процессорной системы с нуля на языке описания аппаратуры SystemVerilog. На базе FPGA (программируемой логической интегральной схемы) последовательно собирается полноценный RISC-V процессор — от простейшего сумматора до процессора с периферией, программируемого на C++.

Инструменты курса:
- **SystemVerilog** — язык описания аппаратуры (не программирования: описываешь схему, а не алгоритм)
- **Testbench** — несинтезируемый модуль для автоматической проверки схемы
- **FPGA / ПЛИС** — кристалл, конфигурацию которого можно менять, реализуя любую цифровую схему
- **Vivado** — САПР от Xilinx: синтезирует SystemVerilog в конфигурацию и прошивает ПЛИС

## Теория

### Зачем нужен сумматор

Сумматор — базовый строительный блок процессора. АЛУ (арифметико-логическое устройство) использует его для всех арифметических операций. В этой лабе собирается 32-битный сумматор из цепочки 1-битных — ровно так, как человек складывает числа в столбик.

### Как работает 1-битный сумматор (fulladder)

Три входа: `a`, `b`, `carry_in` (перенос из предыдущего разряда).  
Два выхода: `sum` (результат), `carry_out` (перенос в следующий разряд).

```
sum     = a XOR b XOR carry_in
carry_out = (a AND b) OR (a AND carry_in) OR (b AND carry_in)
```

`carry_out = 1` когда хотя бы два из трёх входов равны 1 — то есть результат не умещается в один бит.

### Таблица истинности fulladder (carry_in / carry_out)

| a | b | carry_in | sum | carry_out | Пояснение |
|---|---|----------|-----|-----------|-----------|
| 0 | 0 | 0 | 0 | 0 | Ничего не складываем |
| 0 | 0 | 1 | 1 | 0 | Только перенос даёт 1 |
| 0 | 1 | 0 | 1 | 0 | 1 без переноса |
| 0 | 1 | 1 | 0 | 1 | 1+1=2 → перенос |
| 1 | 0 | 0 | 1 | 0 | 1 без переноса |
| 1 | 0 | 1 | 0 | 1 | 1+1=2 → перенос |
| 1 | 1 | 0 | 0 | 1 | 1+1=2 → перенос |
| 1 | 1 | 1 | 1 | 1 | 1+1+1=3 → sum=1, carry=1 |

### Как это выглядит на временной диаграмме (GTKWave)

В официальном тестбенче `lab_01.tb_fulladder.sv` входы задаются так:
- `test_case[2]` → `a`
- `test_case[1]` → `b`
- `test_case[0]` → `carry_in`

То есть test_case проходит значения от 000 до 111, и ты видишь **все 8 комбинаций** входов.  
Для каждой комбинации `sum` и `carry_out` должны соответствовать таблице выше.  
Если на waveform совпадает — сумматор реализован правильно.

### Иерархия модулей

```
fulladder32  (32 бита)
└── fulladder4 × 8  (4 бита каждый)
    └── fulladder × 4  (1 бит каждый)
```

Выходной `carry_out` каждого разряда подаётся на `carry_in` следующего — так реализуется сложение в столбик.

## Пайплайн №1 — Разработка и проверка дома (iverilog + GTKWave)

### Шаг 1. Запуск своего тестбенча с waveform

```bash
iverilog -g2012 -o sim design.sv testbench.sv && vvp sim
gtkwave wave.vcd
```
В GTKWave проверка сигналов:
- Кликни на testbench слева чтобы раскрыть
- Выдели нужные сигналы (например a, b, cin, sum, cout)
- Нажми Append внизу — они появятся на диаграмме
- Нажми Zoom to Fit (кнопка сверху слева)

Что проверяем визуально: каждые 10 ns меняются входы `a`, `b`, `cin`, а выходы `sum` и `carry_out` реагируют мгновенно и корректно.

### Шаг 2. Запуск официальных тестов из репо

```bash
# 1-битный сумматор — перебирает все 8 комбинаций входов
iverilog -g2012 -o sim design.sv lab_01.tb_fulladder.sv && vvp sim
gtkwave wave.vcd
```

```bash
# 4-битный сумматор — перебирает все 512 комбинаций
iverilog -g2012 -o sim design.sv lab_01.tb_fulladder4.sv && vvp sim
gtkwave wave.vcd
```

Успех: в консоли `Test has been finished. Check results at waveform window.`  
Провал: ошибки компиляции — ищи опечатки в именах портов и модулей.

> **Тест для 32-битного (`lab_01.tb_fulladder32.sv`) содержит `assert property` (SVA).**  
> iverilog это не поддерживает — этот тест запускается только в Vivado.

## Пайплайн №2 — Демонстрация в университете (Vivado + ПЛИС)

### Шаг 1. Создать проект в Vivado

1. **File → Project → New**
2. Тип проекта: **RTL Project**
3. Добавить источники: `design.sv` (Design Sources)
4. Добавить тестбенчи: все три `lab_01.tb_*.sv` (Simulation Sources)
5. Выбрать ПЛИС: уточни у преподавателя (обычно Artix-7 или Zynq)

### Шаг 2. Запустить симуляцию

1. В панели **Simulation Sources** правой кнопкой на нужном тестбенче → **Set as Top**
2. **Flow → Run Simulation → Run Behavioral Simulation**
3. В открывшемся окне waveform убедись, что сигналы переключаются корректно

Порядок проверки:
- сначала `lab_01.tb_fulladder` (1 бит)
- затем `lab_01.tb_fulladder4` (4 бита)
- затем `lab_01.tb_fulladder32` (32 бита) — этот запускать только в Vivado

### Шаг 3. Синтез и прошивка

1. **Flow → Run Synthesis** — Vivado превращает SystemVerilog в схему из логических элементов ПЛИС
2. **Flow → Run Implementation** — размещает схему на конкретном кристалле
3. Подключи отладочную плату к компьютеру
4. **Flow → Generate Bitstream** → **Program Device**
5. Плата прошита — можно демонстрировать работу на реальном железе

### Что показывать преподавателю

- Временную диаграмму симуляции всех трёх тестбенчей (нет красных/неопределённых сигналов)
- В TCL-консоли для `tb_fulladder32`: `Number of errors: 0`
- Схему модуля (вкладка **Schematic** после синтеза) — видно как Vivado реализовал сумматор из вентилей
- Работающую прошивку на плате (если требует преподаватель)
