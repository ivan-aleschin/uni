# Лабораторная работа №2 — Арифметико-логическое устройство (АЛУ)

[Теория](THEORY.md)

## Теория

АЛУ (Arithmetic Logic Unit) — это «калькулятор» процессора. На входы поступают два операнда (`a_i`, `b_i`) и код операции (`alu_op_i`), на выходе появляется результат.

В лабе используется **подход мультиплексирования**: несколько операционных блоков работают параллельно, а мультиплексор выбирает нужный результат по коду операции.

### Два независимых выхода

| Выход | Когда активен | Что выдаёт |
|---|---|---|
| `result_o` | Вычислительные опкоды (ADD, SUB, AND...) | 32-битный результат операции |
| `flag_o` | Операции сравнения (EQ, NE, LTS...) | 1-битный флаг условия |

Когда активен `result_o` — `flag_o` всегда равен 0, и наоборот. Именно поэтому удобнее описывать их в двух отдельных `case`.

### Операции

**result_o:**

| Опкод | Действие | Примечание |
|---|---|---|
| `ALU_ADD` | `a + b` | Через сумматор из ЛР1 |
| `ALU_SUB` | `a - b` | Через оператор `-` |
| `ALU_AND` | `a & b` | Побитовое И |
| `ALU_OR`  | `a \| b` | Побитовое ИЛИ |
| `ALU_XOR` | `a ^ b` | Побитовое XOR |
| `ALU_SLL` | `a << b[4:0]` | Логический сдвиг влево |
| `ALU_SRL` | `a >> b[4:0]` | Логический сдвиг вправо |
| `ALU_SRA` | `$signed(a) >>> b[4:0]` | Арифметический сдвиг вправо |
| `ALU_SLTS`| `($signed(a) < $signed(b)) ? 1 : 0` | Знаковое сравнение в число |
| `ALU_SLTU`| `(a < b) ? 1 : 0` | Беззнаковое сравнение в число |

**flag_o:**

| Опкод | Условие |
|---|---|
| `ALU_EQ`  | `a == b` |
| `ALU_NE`  | `a != b` |
| `ALU_LTS` | `$signed(a) < $signed(b)` |
| `ALU_GES` | `$signed(a) >= $signed(b)` |
| `ALU_LTU` | `a < b` |
| `ALU_GEU` | `a >= b` |

### Важно про сдвиги

По спецификации RISC-V используются **только 5 младших бит** операнда `b` — `b[4:0]`. Сдвиг больше чем на 31 для 32-битного числа не имеет смысла. Если не учесть это — процессор будет работать неправильно в дальнейших лабах.

### Знаковые vs беззнаковые операции

По умолчанию SystemVerilog считает все сигналы беззнаковыми. Для знаковой интерпретации используется `$signed()`. Это важно для:
- `ALU_SRA` — `a` интерпретируется как знаковое
- `ALU_SLTS`, `ALU_LTS`, `ALU_GES` — оба операнда знаковые

Пример: число `0xFFFFFFFF` как беззнаковое = 4294967295, как знаковое = -1.

## Про подключение модулей из ЛР1

В SystemVerilog через `import` можно передавать только **параметры, типы и функции** из пакетов — но не модули. Строка `import lab1_32::*` в некоторых реализациях просто игнорируется Vivado.

Модуль `fulladder32` доступен в АЛУ потому, что файл `lab1_adders.sv` добавлен в компиляцию — и дома в командной строке, и в Vivado через Design Sources. Это единственный корректный способ.

## Пайплайн №1 — Разработка и проверка дома (iverilog + GTKWave)

### Шаг 1. Компиляция и запуск

```bash
iverilog -g2012 -o sim alu_opcodes_pkg.sv lab1_adders.sv design.sv testbench.sv && vvp sim
```

> iverilog может выдать предупреждение `sorry: constant selects in always_* processes` — это ограничение самого iverilog, не ошибка в коде. Симуляция проходит корректно. В Vivado это предупреждение не появляется.

Успех: `=== Тест завершён. Ошибок: 0 ===`

### Шаг 2. Просмотр waveform

```bash
gtkwave wave.vcd
```

В GTKWave: раскрой `testbench` → выдели сигналы `op`, `a`, `b`, `result`, `flag` → **Append** → **Ctrl+Shift+F**.

Что проверяем визуально:
- при вычислительных опкодах `result` меняется, `flag` = 0
- при опкодах сравнения `flag` меняется, `result` = 0
- сдвиги: `result` корректно меняется при изменении `b[4:0]`

### Важно про официальный тест

`lab_02.tb_alu.sv` использует `assert property` и `std::randomize` — это не поддерживается iverilog. Тест запускается **только в Vivado**.

## Пайплайн №2 — Демонстрация в университете (Vivado)

### Шаг 1. Создать проект

1. **File → Project → New → RTL Project**
2. **Design Sources**: добавить все три файла:
   - `alu_opcodes_pkg.sv`
   - `lab1_adders.sv`
   - `design.sv`
3. **Simulation Sources**: добавить `lab_02.tb_alu.sv`
4. Выбрать ту же ПЛИС, что и в ЛР1

Именно так Vivado находит `fulladder32` — через наличие `lab1_adders.sv` в проекте, а не через `import`.

### Шаг 2. Порядок компиляции пакета

Если Vivado ругается на неизвестные параметры (`ALU_ADD` и т.д.):
- Вкладка **Compile Order** → правая кнопка на `alu_opcodes_pkg.sv` → **Move to Top**

### Шаг 3. Запустить симуляцию

1. В **Simulation Sources** правая кнопка на `lab_02_tb_alu` → **Set as Top**
2. **Flow → Run Simulation → Run Behavioral Simulation**
3. В TCL-консоли должно быть: `Number of errors: 0`

### Шаг 4. Синтез и прошивка

1. **Flow → Run Synthesis**
2. После синтеза: **Open Synthesized Design → Schematic** — видно структуру мультиплексоров и сумматора
3. **Flow → Run Implementation → Generate Bitstream → Program Device**

### Что показывать преподавателю

- Временную диаграмму симуляции — `result_o` и `flag_o` корректно реагируют на `alu_op_i`
- В TCL-консоли: `Number of errors: 0`
- Схему после синтеза (Schematic)
- Работающую прошивку на плате
