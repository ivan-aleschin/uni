# THEORY — Лабораторная работа №2: АЛУ (Arithmetic Logic Unit)

## 0) Зачем нужна эта тема
АЛУ — центральный «калькулятор» процессора. В ЛР2 ты строишь модуль, который:
- принимает два 32‑битных операнда `a_i`, `b_i`;
- принимает код операции `alu_op_i`;
- выдаёт либо числовой результат (`result_o`), либо флаг сравнения (`flag_o`).

Это базовый блок всех последующих лабораторных, где появляются ветвления и память.

---

## 1) Что такое АЛУ (ALU)
АЛУ — это комбинационная схема, которая выполняет одну из операций:
- арифметика: сложение, вычитание;
- логика: AND, OR, XOR;
- сдвиги: SLL, SRL, SRA;
- сравнения: равенство, меньше/больше.

### Два независимых выхода
В архитектуре курса АЛУ имеет два выхода:
- `result_o` — результат вычислений (32 бита),
- `flag_o` — флаг сравнения (1 бит).

Правило:  
- если операция вычислительная → `result_o` активен, `flag_o = 0`;
- если операция сравнения → `flag_o` активен, `result_o = 0`.

---

## 2) Список операций и смысл

### 2.1 Вычислительные (result_o)
- `ADD`  — сложение (используется сумматор из ЛР1)
- `SUB`  — вычитание
- `AND`  — побитовое И
- `OR`   — побитовое ИЛИ
- `XOR`  — побитовое XOR
- `SLL`  — логический сдвиг влево
- `SRL`  — логический сдвиг вправо (нулевой заполнитель)
- `SRA`  — арифметический сдвиг вправо (знаковый)
- `SLTS` — знаковое сравнение `<` с результатом 0/1
- `SLTU` — беззнаковое сравнение `<` с результатом 0/1

### 2.2 Сравнения (flag_o)
- `EQ`  — a == b
- `NE`  — a != b
- `LTS` — a < b (signed)
- `GES` — a >= b (signed)
- `LTU` — a < b (unsigned)
- `GEU` — a >= b (unsigned)

---

## 3) Числа и знаковость
SystemVerilog по умолчанию трактует сигналы как **беззнаковые**.

### 3.1 Знаковые операции
Для знаковых сравнений и арифметики используется `$signed(...)`.

Например:
- `SRA` должен сохранять знак;
- `SLTS`, `LTS`, `GES` должны сравнивать как signed.

### 3.2 Двоичное доп. кодирование (Two’s complement)
Пример:  
- `0xFFFFFFFF`  
  — unsigned: 4294967295  
  — signed: -1  

Это критично: одинаковый битовый шаблон может означать разное число.

---

## 4) Сдвиги и ограничение b[4:0]
В RISC‑подобной логике используется **только 5 младших бит** операнда `b`:
- `shift = b[4:0]`

Это означает:
- допустимые сдвиги: 0…31;
- сдвиг на 32 и выше не используется (аппаратно обрезается).

---

## 5) Как реализуется ALU внутри
Идея: параллельно вычисляются все кандидаты, потом выбирается нужный по `alu_op_i`.

На практике:
- один `always_comb` для `result_o`;
- второй `always_comb` для `flag_o`;
- для каждого `case` есть `default`.

Это защищает от неопределённых значений (`X`).

---

## 6) Зависимость от ЛР1
`ADD` реализуется через `fulladder32`:
- берётся из файла `lab1_adders.sv`;
- файл должен быть в компиляции (Vivado или iverilog).

---

## 7) Типичные ошибки в ЛР2
- Забыт `b_i[4:0]` при сдвигах → неверное поведение.
- Забыт `$signed(...)` → неверные сравнения.
- Нет `default` → появляются `X`.
- Не добавлен `lab1_adders.sv` → ошибка компиляции.
- Не добавлен `alu_opcodes_pkg.sv` → неопределённые константы.

---

## 8) Мини‑примеры проверки логики

### Пример 1: проверка знакового сравнения
`a = 0xFFFFFFFF`, `b = 1`  
- unsigned: a > b  
- signed: a = -1, значит a < b

Значит:
- `SLTS` даёт 1
- `SLTU` даёт 0

### Пример 2: проверка SRA
`a = 0x80000000`, `b = 1`  
- `SRL` → 0x40000000  
- `SRA` → 0xC0000000 (сохранился знак)

- Чем отличается signed и unsigned.
- Почему используется `b[4:0]`.
- Почему ADD берётся из ЛР1, а не оператор `+`.
