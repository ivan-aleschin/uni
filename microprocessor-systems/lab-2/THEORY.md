# THEORY — Лабораторная работа №2: АЛУ (Arithmetic Logic Unit)

## 0) Зачем эта лабораторная
АЛУ — центральный «калькулятор» процессора. В ЛР2 ты строишь модуль, который:
- принимает два 32‑битных операнда `a_i`, `b_i`;
- принимает код операции `alu_op_i`;
- выдаёт либо числовой результат (`result_o`), либо флаг сравнения (`flag_o`).

Это базовый блок всех последующих лабораторных, где появляются ветвления и память.

---

## 1) Что такое АЛУ (ALU)
АЛУ — комбинационная схема, которая выполняет одну из операций:
- арифметика: сложение, вычитание;
- логика: AND, OR, XOR;
- сдвиги: SLL, SRL, SRA;
- сравнения: EQ, NE, LTS, GEU и т.д.

### Два независимых выхода
АЛУ имеет два выхода:
- `result_o` — результат вычислений (32 бита);
- `flag_o` — флаг сравнения (1 бит).

Правило:
- если операция вычислительная → `result_o` активен, `flag_o = 0`;
- если операция сравнения → `flag_o` активен, `result_o = 0`.

## 2) Таблицы операций (как в README)

### 2.1 Вычислительные (result_o)

| Опкод | Действие | Примечание |
|---|---|---|
| `ALU_ADD`  | `a + b` | Через сумматор из ЛР1 |
| `ALU_SUB`  | `a - b` | Оператор `-` |
| `ALU_AND`  | `a & b` | Побитовое И |
| `ALU_OR`   | `a \| b` | Побитовое ИЛИ |
| `ALU_XOR`  | `a ^ b` | Побитовое XOR |
| `ALU_SLL`  | `a << b[4:0]` | Логический сдвиг влево |
| `ALU_SRL`  | `a >> b[4:0]` | Логический сдвиг вправо |
| `ALU_SRA`  | `$signed(a) >>> b[4:0]` | Арифметический сдвиг вправо |
| `ALU_SLTS` | `($signed(a) < $signed(b)) ? 1 : 0` | Знаковое сравнение в число |
| `ALU_SLTU` | `(a < b) ? 1 : 0` | Беззнаковое сравнение в число |

### 2.2 Сравнения (flag_o)

| Опкод | Условие |
|---|---|
| `ALU_EQ`  | `a == b` |
| `ALU_NE`  | `a != b` |
| `ALU_LTS` | `$signed(a) < $signed(b)` |
| `ALU_GES` | `$signed(a) >= $signed(b)` |
| `ALU_LTU` | `a < b` |
| `ALU_GEU` | `a >= b` |

## 3) Важные нюансы знаковости

### 3.1 Signed vs Unsigned
По умолчанию SystemVerilog трактует `logic [31:0]` как **беззнаковый**.
Чтобы сравнивать как знаковые числа, используют `$signed(...)`.

Пример:
- `0xFFFFFFFF` как unsigned = 4294967295
- `0xFFFFFFFF` как signed = -1

### 3.2 Где нужен `$signed(...)`
- `ALU_SRA`
- `ALU_SLTS`, `ALU_LTS`, `ALU_GES`

---

## 4) Сдвиги и `b[4:0]`
По спецификации используется только **5 младших бит** операнда `b`:
- `shift = b[4:0]`

Это значит:
- допустимые сдвиги 0…31;
- сдвиг на 32 и выше игнорируется (обрезается).

## 5) SystemVerilog‑синтаксис, который ты реально используешь

### 5.1 Объявление модуля и портов
Пример: `module alu (input logic [4:0] alu_op_i, input logic [31:0] a_i, b_i, output logic [31:0] result_o, output logic flag_o);`

Что это значит:
- `logic [31:0]` — 32‑битный сигнал
- `logic` подходит и для wire, и для reg
- входы/выходы описываются прямо в заголовке модуля

### 5.2 Импорт пакета с опкодами
`import alu_opcodes_pkg::*;`
Это даёт доступ к именам `ALU_ADD`, `ALU_SUB` и т.д.

### 5.3 Комбинаторная логика
Используется `always_comb`:
- нет `posedge`
- выходы меняются сразу при изменении входов

### 5.4 `case` и `default`
Всегда должен быть `default`, иначе могут появиться `X` в симуляции.

## 6) Разбор `design.sv`: что и зачем

### 6.1 Сумматор из ЛР1
Внутри АЛУ добавляется:
- `fulladder32` для операции ADD
- `add_result` — хранит сумму
- `add_carry` — не используется, но порт нужно подключить

Логика:
- `ALU_ADD` берёт `add_result`
- остальные операции идут отдельными выражениями

### 6.2 `result_o` (вычислительные операции)
В `always_comb` на `result_o`:
- сначала `result_o = '0` (обнуление)
- потом `case (alu_op_i)` с вычислениями

### 6.3 `flag_o` (операции сравнения)
В отдельном `always_comb`:
- сначала `flag_o = 1'b0`
- потом `case (alu_op_i)` с условиями

Это делает `result_o` и `flag_o` независимыми.

## 7) Типичные ошибки
- забыли `b[4:0]` → неверные сдвиги
- забыли `$signed(...)` → неверные сравнения
- нет `default` → появляются `X`
- нет `lab1_adders.sv` в компиляции → ошибка
- не импортирован пакет опкодов
