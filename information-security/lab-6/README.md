## 2. Краткая теория

Парольная защита — один из базовых методов аутентификации. При проектировании системы парольной защиты важно учитывать:

- Надёжное хранение паролей:
  - Никогда не хранить пароли в открытом виде.
  - Использовать соль + итерационное хеширование (PBKDF2, bcrypt, scrypt, Argon2).
  - Соль должна быть уникальной для каждой учётной записи и храниться в БД вместе с хешем.
- Устойчивость к перебору:
  - Ограничение числа попыток входа; временная блокировка или увеличение задержки после неудач.
  - Распознавание подозрительной активности (браузер/IP, множественные неудачи).
- UI/UX для пароля:
  - Не отображать вводимые символы.
  - Поддержка требований сложности (минимальная длина, набор символов).
- Методы аутентификации:
  - Полный ввод пароля;
  - Выборка символов (вариант вашей работы): система запрашивает отдельные позиции пароля;
  - Одноразовые пароли (OTP);
  - Challenge–response (запрос–ответ) и схемы с хендшейком;
  - Биометрия, смарт-карты и др.
- Дополнительно:
  - Логи входов/ошибок, оповещения администратора;
  - Принудительная смена пароля по сроку;
  - Защита от перехвата ввода (анти-кейлоггер меры на уровне ОС, аппаратуры — за рамками лабораторной).

## 3. Выбранный метод (вариант 1 — выборка символов)

Описание метода:
- При регистрации пользователь создаёт пароль `P` длины `L`. Система генерирует уникальную соль `salt` и сохраняет значение `H = Hash(password, salt)` (итерационный PBKDF2/HMAC-SHA256 и число итераций).
- При аутентификации система случайно выбирает `k` позиций из диапазона `1..L` (например, 2 позиции: 2-я и 5-я) и запрашивает у пользователя символы на этих позициях.
- Пользователь вводит требуемые символы, система проверяет соответствие введённых символов ожидаемым, не требуя ввода всего пароля.
- Проверка корректности символов реализуется путём сравнения: для обеспечения безопасности хранить нельзя явные символы позиций, поэтому возможны два подхода:
  1. Хранить хеши отдельных символов/позиции вместе с солью (менее предпочтительно — раскрытие структуры пароля).
  2. На момент проверки запрашивать весь пароль (в реальной системе) и сверять хеш — но в лабораторном варианте допустимо хранить хеши отдельных требуемых подстрок или использовать протокол challenge–response.
- Для учебной реализации: храним полный хеш пароля + соль, но при проверке попросим пользователя ввести весь пароль в скрытом режиме для вычисления ожидаемых символов (в отчёте нужно отметить это упрощение) — или реализовать хранение контрольных хешей отдельных позиций (см. замечания безопасности в отчёте).

Защита от перебора:
- Максимум попыток `MAX_ATTEMPTS` (например, 3);
- Блокировка аккаунта на `LOCK_TIME_SECONDS` при превышении попыток;
- Логирование событий.

## 4. Алгоритм работы программы (пошагово)

1. При первом запуске можно предложить зарегистрировать пользователя:
   - Ввод имени пользователя;
   - Ввод пароля (без эха);
   - Генерация `salt` (например, 16 байт, `secrets.token_bytes(16)`), вычисление `hash = PBKDF2(password, salt, iterations=100_000, dklen=32, hash=sha256)`.
   - Сохранение записи: `{username, salt (hex), hash (hex), iterations}` в `users.json`.
2. Для входа:
   - Ввод имени пользователя;
   - Если учётная запись заблокирована — сообщить и завершить;
   - Система выбирает случайно `k` позиций (например, 2 позиции) и отображает их (1-based): `Введите символы на позициях 2, 5`.
   - Пользователь вводит требуемые символы (без отображения).
   - Вариант проверки A (упрощённый, удобный для лаборатории):
     - Система запрашивает полный пароль (скрыто) и проверяет `PBKDF2(password, salt) == hash` (более безопасный способ на практике).
     - Или: для демонстрации — сравнивает введённые позиции с соответствующими символами из введённого полного пароля.
   - При ошибке увеличивается счётчик попыток; при превышении `MAX_ATTEMPTS` — блокировка.
3. При успешной аутентификации:
   - Вывести дату и время последнего входа (если хранится) и обновить значение последнего входа.
4. Дополнительно:
   - Команды администратора для сброса блокировок.
   - Экспорт/импорт user-файлов (только в зашифрованном/хешированном виде).

## 6. Пример псевдо-/приближённого кода (порядок операций)

Пример структуры `main.py` (псевдокод, для отчёта и реализации):

```uni/information-security/lab-6/main.py#L1-L120
# Псевдокод: структура программы
import os, json, time, secrets, getpass
from hashlib import pbkdf2_hmac

# Константы
MAX_ATTEMPTS = 3
LOCK_TIME_SECONDS = 300
K_SELECTED = 2
ITERATIONS = 100000

# Регистрация пользователя
def register(username):
    password = getpass.getpass("Введите пароль: ")
    salt = secrets.token_bytes(16)
    hash = pbkdf2_hmac('sha256', password.encode(), salt, ITERATIONS)
    save_user(username, salt.hex(), hash.hex(), ITERATIONS)

# Аутентификация (вариант с выборкой символов)
def authenticate(username):
    user = load_user(username)
    if is_blocked(user): reject()
    # выбрать k случайных позиций
    positions = select_positions(len_estimated_password, K_SELECTED)
    prompt_user_for_positions(positions)  # UI: показать номера позиций
    # Для лабораторного упрощения попросим пользователя ввести весь пароль скрыто:
    password = getpass.getpass("Введите весь пароль (скрыто, для проверки): ")
    hash_try = pbkdf2_hmac('sha256', password.encode(), bytes.fromhex(user['salt']), user['iterations'])
    if hash_try.hex() == user['hash']:
        success()
    else:
        fail_and_update_attempts()
```

Примечание: в реальной системе не следует запрашивать полный пароль, если цель — выборка символов; в лабораторной реализации это упрощение допускается для демонстрации логики и проверки хеширования. В отчёте опишите это допущение и варианты улучшения (challenge–response, хранение хешей отдельных позиций с солью и т.п.).
